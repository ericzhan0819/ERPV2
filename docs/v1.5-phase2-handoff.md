## 實作內容

- 新增 `UsernameRules`，集中 nullable、3～30 字元、字元白名單、application unique rule 與繁體中文訊息。
- 驗證前只正規化字串或 null；array 等錯誤型別保留給 Validator 回報，不會被轉成空值而誤放行。
- 新增 `UpdateCurrentUserProfileRequest` 作為第 7 部分 self profile API 的正式 Request 契約；目前尚未掛 route，但已實測使用同一組 normalization、unique rule 與 current-user ignore。
- `UserService::setUsername()` 保留 model mutator 的 trim／小寫／空字串轉 null 邊界，並將 MySQL／SQLite 的 username duplicate-key 轉成 `username` validation error。
- 真實 MariaDB 測試使用兩個 process、兩條獨立連線與未提交 winner transaction，確認 loser 會等待 unique index，winner 提交後只有一筆成功，loser 收到 validation error 而非 500。

## 重要契約

- Request 在套用 `UsernameRules::rules()` 前，必須先用 `UsernameRules::normalizeInput()` 正規化 username。
- Eloquent 寫入仍由 `User` username mutator 作最後正規化防線，Factory 與 Service 不得繞過。
- Application unique rule 提供一般 422；database unique index 才是競態時的最終正確性保護。
- 寫入 username 的正式 Service 路徑必須使用 `UserService::setUsername()`，讓 duplicate-key race 維持欄位型 validation error。

## 後續邊界

- 第 3 部分調整 `LoginRequest` 與 `AuthService` 時，沿用 `User::normalizeUsername()`；不得把 profile 的 unique rule 套到登入輸入。
- 第 4 部分 Rate Limiter 必須讓同一 User 的 username／Email 共用 canonical account key。
- 第 7 部分建立 self profile Controller／Service 時，直接掛用 `UpdateCurrentUserProfileRequest`，並補齊禁止修改其他欄位的 fail-closed 規則。
