## 4. Smoke 過程結論

首次 Firefox 驗證沒有發現需要修改產品程式碼的缺陷。過程中數次中止均為驗證腳本 selector 或測試資料排序假設過嚴：

1. Mobile 篩選按鈕包含作用中條件數，不能用完整文字等於「篩選」定位。
2. 新增分頁資料後第一頁由較新的 `PagedBrand` 佔滿，不能用固定品牌文字判斷卡片載入。
3. Drawer 與頁面同時存在「清除全部」，必須限定在 `role=dialog` 內操作。
4. checkbox 的實際觸控區是外層 44px label，不能以 16px 原生 checkbox 圖形尺寸判定觸控失敗。

上述均已以正確 DOM／互動邊界重驗通過，沒有改動應用程式或正式資料。

### 4.1 iPhone 初驗發現與修正

使用者以真實手機確認中文輸入法正常，但 iPhone Safari 在主題切換與載入時偶爾不順暢，瀏覽器上下 Safe Area 會停留舊灰色，必須刷新才更新。查證後確認既有主題直到 React `useEffect` 才套用，首次繪製固定使用淺色 `theme-color`；切換時也只更新既有 meta content 與 CSS 變數，無法可靠要求 Safari 重繪瀏覽器上下區域。

最小修正：

- 在 `index.html` 的 `<head>` 於首次繪製前讀取已保存／系統主題，同步 `html.dark`、`color-scheme`、背景與 `theme-color`。
- React 改以 `useLayoutEffect` 同步 `html`／`body`／`#root` 的實際底色，避免主題更新後 viewport 背景仍停留舊值。
- 主題變更時重新掛載 `theme-color` meta，要求 iOS Safari 重繪瀏覽器上下區域。
- 在 BFCache `pageshow`、重新顯示與旋轉時再次同步主題及底色。
- `localStorage` 讀寫加入 Safari 儲存空間受限時的降級，不讓主題初始化阻斷載入。

修正後驗證：

```text
npm test                 PASS：3 files／14 tests
npm run lint             PASS：0 errors；2 個既有 Fast Refresh warnings
npx tsc -b --noEmit      PASS
npm run build            PASS：1890 modules
Firefox browser regression：30 checks passed
```

Firefox 回歸涵蓋 light／dark 首次載入、連續切換 6 次、`pageshow`、旋轉事件，以及 320／375／390／768／1440px 的 `html`／`body`／`#root`／`theme-color` 同色與無水平 overflow。真實 iPhone Safari repaint 仍須由使用者複驗，不能以 Firefox 取代。

使用者第一次複驗後進一步確認：直接切換主題已改善，但 Mobile Sidebar 開啟再收回時，Safari 上下區域仍可能停留 overlay 的反灰狀態。查證後確認 Sidebar cleanup 只恢復 `body` overflow 與 focus；overlay 移除後沒有再次要求 Safari 重繪瀏覽器底色。

第二次最小修正：

- `ThemeProvider` 提供穩定的 `refreshTheme()`，統一重新套用背景並掛載 `theme-color` meta。
- Mobile Sidebar cleanup 在恢復 overflow、移除 overlay 後的下一個 animation frame 呼叫 `refreshTheme()`。
- 關閉按鈕、`Escape`、點擊 overlay、點導覽換頁都共用同一 cleanup，不建立平行關閉邏輯。

修正後以 Firefox 在 light／dark 各自故意污染 `html`／`body`／`#root`／`theme-color`，再走上述四種關閉方式；12 項重繪與操作檢查全數通過。真實 iPhone Safari 仍維持待使用者複驗。

使用者第二次複驗確認上述重繪呼叫仍無法讓 Safari 恢復，上下底色必須手動刷新。進一步查到這不是 theme state 遺漏，而是 iOS 26 WebKit 已知的 fixed overlay scroll pocket tint 問題：`position: fixed` 元素的半透明背景會取代上下瀏覽器區域顏色，且實際網站可能在 overlay 關閉後仍保留舊色。WebKit 官方紀錄見 [bug 300965](https://bugs.webkit.org/show_bug.cgi?id=300965) 與 [bug 302272](https://bugs.webkit.org/show_bug.cgi?id=302272)。

第三次修正改為避開觸發條件：

- fixed overlay 本身固定保持透明，不再使用 `bg-black/50`。
- 改用 `backdrop-filter: brightness(0.5)` 壓暗 overlay 後方頁面；WebKit 同一重現紀錄指出 backdrop-filter 不會把顏色延伸到上下區域。
- 不支援 backdrop-filter 的非現代瀏覽器才使用半透明背景 fallback。
- 移除上一版無效的 Sidebar → ThemeProvider 重繪 coupling；主題首次載入、切換、BFCache 與旋轉同步仍保留。

第三次修正後 frontend test／lint／typecheck／build 維持通過。Firefox 在 light／dark、四種 Sidebar 關閉方式下確認 fixed overlay computed background 均為透明、backdrop filter 均為 `brightness(0.5)`、`theme-color` 從開啟到關閉都不改變，合計 32 項檢查通過。真實 iPhone Safari 仍須再次複驗。

使用者第三次複驗指出透明 fixed overlay 雖避開染色，卻讓 light mode 的 Safari 上下區域不反灰，與頁面中央的暗化效果形成突兀白邊；而 Sidebar 曾開合後再切換 light／dark，上下區域仍可能卡在切換前顏色。這證明只調整 fixed overlay 的背景方式無法同時滿足開啟與關閉狀態。

第四次修正改為完整避開 fixed tint 機制並明確管理 viewport tint：

- Mobile Sidebar 與 overlay 從 `position: fixed` 改為 App Shell 內的 `position: absolute` viewport-height layer；Desktop 仍為既有 `static` Sidebar。
- overlay 恢復 50% 黑色背景，但因不再是 fixed 元素，不觸發 WebKit fixed scroll pocket tint。
- Sidebar 開啟時，`ThemeProvider` 將 `html`／`body`／`#root`／`theme-color` 同步為目前主題疊加 50% 黑色的精確合成色：light `#7C7D7E`、dark `#060910`。
- Sidebar 關閉時恢復正式 light `#F8FAFC` 或 dark `#0B1120`；之後主題切換仍由同一同步函式更新，不保留 Sidebar tint。
- 關閉按鈕、`Escape`、點 overlay 與導覽換頁皆共用同一狀態與 cleanup。

第四次修正後 frontend test／lint／typecheck／build 通過；Firefox 54 項狀態序列檢查涵蓋四種關閉方式的「light → 開啟反灰 → 關閉恢復 → dark → light」、dark 開啟／關閉，以及 Desktop static Sidebar。另於 320×568、375×667、390×844、844×390 橫向與 768×1024 驗證 Sidebar／overlay 高度等於 visual viewport、沒有水平 overflow，開啟／關閉 tint 正確，共 10 項通過。使用者於 2026-07-22 以真實 iPhone Safari 完成最終複驗，確認 Sidebar 開啟時上下區域自然反灰、關閉後恢復，且後續 light／dark 切換不再卡住舊色或需要手動刷新。
