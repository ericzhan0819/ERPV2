# ERPV2 v1.5 第 4 部分交接

日期：2026-07-24

## 完成範圍

- 正式完成 `PLAN_v1.5.md` 第 4 部分「Login Rate Limiter 雙識別安全」的逐項驗收。
- 核心 canonical limiter 已因第 3 部分 adversarial review 發現 alias bypass 而提前修正；本輪未重造第二套 limiter。
- 補強不存在 username／Email 混合輪替與 Email 成功登入清除共用 alias 額度的直接回歸證據。
- 未進入第 5 部分首次登入與 Admin 重設密碼狀態。

## 正式實作契約

- IP-wide limiter 保持獨立，固定只累積失敗認證；已封鎖 IP 會在解析 User 前回 429。
- identifier + IP 與 account limiter 對可解析帳號都使用 `uid:<user id>`，所以 username、Email 與大小寫變體共用同一額度。
- 不存在或 Email 大小寫資料不唯一的識別字使用正規化字串 SHA-256，key 穩定、不含原始 PII，且不同識別字不互相污染。
- identifier + IP 每 60 秒最多 5 次；account 每 15 分鐘最多 10 次；IP-wide 每 60 秒最多 30 次失敗認證。
- identifier + IP 與 account 在 credential check 前以 atomic `RateLimiter::hit()` 占用額度；超限請求不再執行 `Auth::attempt()`。
- 成功登入只清除該 User 的 canonical identifier + IP 與 account limiter，不清除其他 User 的額度，也不累積 IP-wide failure。
- 429 固定保留 `Retry-After`；帳密錯誤仍使用通用訊息，不洩漏 identifier 是否存在。

## 本輪補強

- `LoginThrottleTest` 的 IP-wide rotating identifier 測試改為交錯不存在 username 與 Email，直接鎖定兩種識別格式都不能繞過同 IP 額度。
- 新增 Email 成功登入案例：先以 username／Email alias 累積九次 canonical account failure，再用 Email 成功登入，確認後續 username 仍取得清除後的新額度。
- 既有 username 成功登入清除 Email alias 額度、正反 alias 累積、大小寫變體、identifier + IP、rotating IP、IP-wide、封鎖後不認證、成功不清除他人額度等測試全部保留。

## 已知取捨

- Canonical limiter 必須先把 identifier 解析為 User ID。因此 IP-wide gate 之後，identifier + IP 或 account 已封鎖的 Email 請求仍會執行一次 `LOWER(email)` 查詢。
- ERPV2 是小型內部系統，目前接受此低成本查詢。若未來要完全消除，需另立 Email 小寫正規化、既有資料回填、case-variant 重複資料處置與索引 migration 計畫，不屬第 4 部分。
