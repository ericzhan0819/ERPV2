# v1.3 第 5 部分交接：整月跨級獎金計算器

日期：2026-07-13

## 完成範圍

`PLAN_v1.3.md` 第 5 部分已完成：

- 新增無寫入副作用的 `SalaryCommissionCalculator`，集中處理薪資獎金公式。
- 批次只讀取傳入車輛的 approved MoneyEntry；pending／rejected 不計入。
- 單車計算回傳收入、支出、毛利、公司營運保留、分配池、收車獎金、賣車獎金、公司剩餘與公司最終取得。
- 毛利小於或等於 0 時保留原毛利供明細顯示，但所有自動獎金皆為 0，不產生負獎金。
- 同一批 eligible 車輛依 `sales_agent_id` 分組，從方案 tiers 套用整月跨級比例；第 5 台成立後，同一人的前 4 台會一併使用 50%。
- 回傳每位賣車人的符合資格台數、目前級距、距下一級台數、目前賣車獎金合計與既有車輛跨級後預估增加金額。
- basis points 計算全部向下取整，餘數由公司剩餘分配額吸收，正毛利分配總和恆等於毛利。
- 比例乘法使用商數／餘數拆算，避免大額 `amount × bps` 中間值 overflow；彙總若超過 PHP 安全整數範圍會拋出明確例外。
- 混合盈虧批次回傳 `loss_total` 與 `company_net`。`company_total` 是獲利車分配給公司的正額合計；正式公司淨得必須使用 `company_net`，且批次層以硬性 guard 保證 `company_net + purchase_bonus + sales_bonus = gross_profit`。
- `calculate()` 必須傳入 `YYYY-MM` 結算月份及啟用佣金的 agent ID 集合。每台 `sold_at` 必須屬於該月；未啟用佣金的人員依收車／賣車角色個別套用 0 bps，該金額自然回到公司剩餘，不會把整台車踢除或吞掉另一位人員獎金。
- approved 收支聚合透過 `MoneyEntry::approved()` scope，避免正式薪資規則與其他金額彙總漂移。
- 所有薪資月份參數共用 `SalaryPeriodMonth` 的 `YYYY-MM` 契約；方案 scope 會自行轉為月首日再查詢，避免 SQLite／MySQL 日期隱式轉型差異。
- `CommissionPlanService::findEffectiveForMonth()` 負責建立草稿時的正式方案選取；計算器只要求傳入方案已儲存且 tiers 合法。既有 SalaryPeriod 重算必須沿用自己綁定的 `commission_plan_id`，不因後來新增回溯生效方案而換版或阻斷。
- `SalaryProfile.user_id` 已 cast 為 integer，佣金資格集合不依賴 PDO driver 型別。

## 計算器邊界

第 5 部分只負責公式，不負責完整決定車輛資格。呼叫端必須傳入已通過資格檢查的車輛集合、結算月份與從 active Salary Profile 產生的 `commission_enabled=true` agent IDs；計算器不會靜默略過傳入資料，也不會推定收車人／賣車人。第 6 部分需新增集中資格服務，檢查 sold 狀態、sold_at 月份、收付款完整性、pending／legacy_unknown、歸屬與跨月份重複引用，再把合格集合交給計算器。

同一人兼任收車人與賣車人時，逐車結果保留兩個 agent id 與兩種獎金；後續建立 settlement items 時可各自歸入該員工。不同人時則依各欄位分開歸屬。

使用者已鎖定：虧損或零毛利但其他資格完整的成交車仍保留在月份明細與公司淨損益，但不計入賣車跨級台數。只有 `gross_profit > 0` 的車輛增加 `eligible_sales_count`；計算器已在取得 approved-only 毛利後再分組計數並有 4 台正毛利 + 零毛利／虧損車的回歸測試。

`sales_agents` 必須先以本批次出現且啟用佣金的所有賣車人建表，正毛利成交台數另行填入；整月只有虧損車或只有零毛利車的業務仍回傳 0 台、0 bps、0 元。不得再從 profitable counts 直接建立 map，否則會重現 undefined key fatal crash。

全系統業務日期與月份邊界已統一為 `Asia/Taipei`：Laravel app timezone 使用 `Asia/Taipei`，MySQL／MariaDB session 使用 `+08:00`，API datetime 保留 `+08:00` offset，前端預設日期、列印與稽核顯示固定以台北時間格式化。Dashboard 與薪資共用台北月首邊界。既有 MySQL／MariaDB `TIMESTAMP` 由 session timezone 轉換，不做資料搬移；持久化 SQLite 舊 UTC-naive 開發資料需重建。

上述「既有 TIMESTAMP 不搬移」只適用於切換前資料庫實際使用 UTC 的環境。部署前必須在舊版本連線執行：

```sql
SELECT @@global.time_zone, @@session.time_zone, @@system_time_zone, NOW(), UTC_TIMESTAMP();
```

只有 global/session 明確為 UTC，或 `SYSTEM` 且 system timezone 為 UTC、`NOW()` 等於 `UTC_TIMESTAMP()` 時，才可直接切換。若不是 UTC，停止部署並人工評估既有 TIMESTAMP，禁止猜測式批次平移。

`TimezoneMysqlIntegrationTest` 使用安全 allowlist 與可拋棄 test schema，透過兩個獨立 PDO connection 驗證台北 `2026-07-01 00:10` 的 MySQL TIMESTAMP epoch 對應 UTC `2026-06-30 16:10`；SQLite suite 只會 skip，不冒充真實資料庫覆蓋。
