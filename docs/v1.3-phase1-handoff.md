## 2. 重要資料安全設計

### 獎金方案

- 比例只使用整數 basis points。
- `CommissionPlanRules` 驗證至少一個 tier、第一級從 1 台開始、台數遞增、排序不重複、比例 0～10000、收車加任一賣車級距不超過 100%。
- 一旦方案被 `salary_periods` 引用，DB trigger 會阻擋計算欄位與 tiers 的新增／修改／刪除。
- `CommissionPlanSeeder` 可重跑；未使用方案可修復成標準值，已使用方案只核對、不覆寫歷史。

### 歷史車輛歸屬

- migration 只新增 nullable FK，沒有 `UPDATE` 或 heuristic backfill。
- 不使用 `created_by`／`updated_by` 推定實際收車人／賣車人。
- 歷史已售車輛待後續第 4.3 部分由 admin 人工補齊。

### 薪資 MoneyEntry

- DB trigger 限制 `salary_settlement` 只能由 admin 建立，且必須是 `expense`、`薪資 / 佣金`、`vehicle_id=null`、`approval_status=approved`。
- 現有一般 Money Entry CRUD 對所有非 `manual` 來源一律拒絕 update/delete，因此新來源自動受保護。
- `APPROVABLE_SOURCE_TYPES` 不含新來源，不能走一般 approve/reject。
- manager／sales 在列表與單筆 Policy 都無法取得薪資來源；Resource 再額外移除薪資金額、帳戶、員工姓名、說明與審核欄位。

## 3. Migration 邊界

本階段 migration 為 `2026_07_13_000000` 至 `000010`，拆分成可獨立記錄與 rollback 的小步。`000008`、`000010` 是明確的 forward migration：開發 MariaDB 驗證期間分別補上跨資料庫 agent indexes／月份首日 constraint，以及 salary MoneyEntry 的 DB 級不可變保護，因此沒有回改已執行 migration。

- `up` 不清空、不改寫既有營運資料。
- `down` 依相反順序移除 trigger、constraint、薪資表與新增 FK。
- 若中途失敗，Laravel 只會記錄已成功的 migration；修正後可從未完成步驟續跑。
- 已有薪資資料的環境不應任意 rollback；尤其移除 salary tables／agent 欄位會刪除 v1.3 資料，執行前需備份與人工確認。

開發 MariaDB 已以增量方式執行 `php artisan migrate` 與 `CommissionPlanSeeder`，沒有執行 `migrate:fresh` 或 `db:wipe`。
