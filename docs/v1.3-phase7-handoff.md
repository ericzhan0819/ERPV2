# v1.3 第 7 部分交接：薪資草稿、重算與確認

日期：2026-07-14

## 完成範圍

`PLAN_v1.3.md` 第 7 部分已完成：

- 新增 `SalaryPeriodService`，集中處理月份草稿建立、重算、手動加扣與確認鎖定。
- 每月只允許一筆 SalaryPeriod；建立草稿時才由 `findEffectiveForMonth()` 選取當下最新有效方案，沒有方案或月份重複皆回 422。
- 只有 active user + active salary profile 建立 settlement。active user 若沒有啟用薪資設定採明確排除，不以零薪資靜默納入。
- 草稿建立／重算會複製目前薪資設定、重建底薪／津貼／勞健保／收車獎金／賣車獎金項目，並由 item 彙總 gross、deduction、net。
- 重算固定沿用 SalaryPeriod 已綁定的 `commission_plan_id`；後來新增回溯生效方案不會換版。整月成交台數變動時，同一賣車人的所有車輛重新套用最高級距。
- 手動加扣只允許 `manual_addition`／`manual_deduction`、正整數金額與必填說明；重算不刪手動項目，自動項目不可從手動刪除入口修改。
- 確認在單一 transaction 內鎖 period 與整月候選車、重跑資格、重算公式並比對草稿 snapshot；資料過期時回 422 要求先重算，不會直接確認 stale preview。
- 確認成功寫入 `confirmed_by`／`confirmed_at`。confirmed／paid 後，重算與手動加扣一律拒絕。
- `closeSale()` 回填到 confirmed／paid 月份時回 422，訊息明確要求聯絡管理員；draft 月份仍可成交後重算。
- 已售車歸屬除了既有 item reference，另依 `sold_at` 月份防禦 confirmed／paid 歷史缺口，零獎金或早期沒有 item 的車也不能改歸屬。
- 月份建立／重算／確認及手動加扣新增／刪除會寫 Audit Log，但不把薪資金額、加扣金額或說明複製到 audit metadata。

## 重要契約

- 第 7 部分只完成服務層與跨模組鎖定；SalaryPeriod Policy、Request、Resource 與 API routes 明列於 PLAN 第 9、10 部分，尚未提前開放。
- `inspectPeriodForUpdate()` 只能在 transaction 內呼叫，草稿可保留異常但會鎖定完整月份候選車；`assertPeriodEligible()` 仍是確認時唯一入口，異常會阻擋。
- Salary Profile 在草稿期間修改後，下一次重算會更新 snapshot。若既有 settlement 的 profile／user 後來停用，自動項目歸零，但手動項目保留，交由 admin 明確處理。
- `salary_settlements.net_pay` 使用 signed bigint；草稿可保存負薪，避免單一純佣金／離職員工讓整月建立或重算 rollback。admin 可新增手動加給補平或刪除扣款，確認時才列出仍為負薪的員工並回 422。
- 新增手動扣款若當下會造成負薪仍立即回 422 並 rollback；重算造成的負薪則必須保留在草稿，不能用同一檢查癱瘓整月。
- 獎金 item 的 `calculation_snapshot` 保存 approved 收入／支出、毛利、公司保留、分配池、收／賣車比例與金額、公司剩餘、人員歸屬及月份。confirmed／paid 後只讀這些 snapshot，不重新選最新方案。
- `closeSale()` 採 period → vehicle 鎖順序；薪資建立／重算／確認也先鎖 period 再鎖候選車，避免回填成交與確認形成反向鎖。
- 月份鎖使用 `period_month = YYYY-MM-01` 等值查詢以命中 MySQL unique index；SalaryPeriod model 固定以 `Y-m-d` 寫入，避免 SQLite DATE 欄保存時間部分。
- 建立草稿以 locking read 固定方案；重算與確認皆先鎖候選車，再讀 MoneyEntry、snapshot 與 CommissionPlan，避免在 MySQL REPEATABLE READ 下提早建立舊 read view。
