## 2. API 與權限

已完成：

```text
GET  /api/salary-profiles
PUT  /api/salary-profiles/{user}
GET  /api/commission-plans
POST /api/commission-plans
GET  /api/commission-plans/{commissionPlan}
```

全部位於 `auth:sanctum` + `active` + `role:admin`，並有對應 Policy 白名單。`manager`、`sales`、未知角色皆為 403。Resource 採白名單；薪資設定 API 不回傳 settlement totals 或 snapshot。

## 3. 重要規則

- 薪資設定四個金額皆為非負整數，前端不得傳入 user id、snapshot 或 totals。
- inactive user 可保留 `is_active=false` 的設定，但不可建立／切換為 active profile。
- tiers 至少一級、第一級從 1 台開始、台數嚴格遞增、比例為 0～10000 bps、收車加任一賣車比例不可超過分配池 100%。
- `sort_order` 由後端依 tiers 陣列順序產生，不信任前端排序值。
- 月份選取 active 且 `effective_from <= period_month` 的最新生效方案；同日以較新 id 決定。
- App API 不提供方案 update／delete；已引用方案另有 Phase 1 DB trigger 保護，避免繞過應用層回改歷史。

## 5. 審查修正

- `SalaryProfileService::upsertProfile()`：首次建立撞 `salary_profiles.user_id` unique constraint 時，讓原 transaction 完整 rollback，再以 fresh transaction／locking read 取得 winner。相同 payload replay；不同 payload 回傳 `422 user`。winner 不存在或不是指定 constraint 時保留原 QueryException，不誤吞其他 DB 問題。
- `CommissionPlanService::createPlan()`：request unique 驗證後仍撞同名 race 時，rollback 後以 fresh transaction／locking read 確認 winner，再回傳 `422 name`；方案與 tiers 不會留下半套資料或重複 audit。
- `SalarySettingsMysqlConcurrencyTest` 以 `pcntl_fork` 建立真正父子 PHP 請求、socket barrier 固定時序、各自 `DB::purge()` 後建立獨立 MariaDB connection，直接呼叫 `upsertProfile()`／`createPlan()` 公開方法，確認輸家確實走完整 try/catch。測試包含 APP_ENV、runningUnitTests、connection allowlist、database allowlist 與可拋棄資料庫名稱安全閘門；實測 schema 已在驗證後刪除。
