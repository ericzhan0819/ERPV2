## 2. URL 與 API 契約

Vehicle 工作區 URL：

```text
/vehicles
→ 前端預設 preparing,listed,reserved

/vehicles?status=preparing
→ 單一狀態相容

/vehicles?status=preparing,listed&is_preparation_completed=false&page=2
→ 多狀態、整備未完成、第 2 頁
```

API 另接受 `status[]=preparing&status[]=reserved`。未傳 `status` 時 API 仍不限制狀態；三狀態預設是 Vehicle 工作區契約，不會改變其他 API 呼叫端。

MoneyEntry 工作區以 `approval` 保存分享 URL，呼叫 API 時映射為既有 `approval_status`：

```text
/money-entries?approval=pending&direction=expense&page=2
```

這不改變後端 MoneyEntry Request／Service、CRUD、approval 或角色遮蔽。

## 4. Adversarial review 修正

逐項查證 review 的 1 項 Medium 與 4 項 Low findings，5 項均有效並完成最小修正：

- `is_preparation_completed=` 經 Laravel 正規化為 `null` 時不再被 `filter_var()` 誤判為 `false`；Service 只在值非 null 時套用條件，並補空值等同未篩選的回歸測試。
- MoneyEntry category 依還原後的 direction 合法清單驗證，不相容或未知 category 不送 API。
- MoneyEntry 的 active Filter 判斷改看正式篩選欄位，不把單獨 page 視為篩選。
- 預設 Vehicle 狀態集合 predicate 集中於 `listFilters.ts`，URL 序列化與頁面提示使用同一判斷。
- MoneyEntry 列表加入遞增 request sequence；只有最新請求能更新 entries、meta、error 與 loading，元件 cleanup 也會使未完成請求失效。

## 5. 後續入口

下一階段依 `PLAN_v1.4.md` 第 3 部分實作 Dashboard API。Dashboard KPI 導流應直接使用本階段契約：

- 待整備：`/vehicles?status=preparing&is_preparation_completed=false`
- 待上架：`/vehicles?status=preparing&is_preparation_completed=true`
- 待交車：`/vehicles?status=reserved`
- 待審核：`/money-entries?approval=pending`

不得在 Dashboard 建立第二套 Filter state；第 4、6、7 部分應持續沿用同一組 parse／serialize 函式。
