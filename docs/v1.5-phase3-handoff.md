# ERPV2 v1.5 第 3 部分交接

日期：2026-07-24

## 完成範圍

- 完成 `PLAN_v1.5.md` 第 3 部分 Login Request 與 Email／username 雙識別登入。
- 更新 Auth API 文件、`UserResource` 輸出與登入相關自動測試。
- 對立性審查確認雙識別會稀釋既有限流後，依既有 PLAN §4 完成 canonical Rate Limiter 與 alias bypass 回歸，沒有另創安全設計。
- 前端 API 已最小切換為送出 `login` payload，恢復 Email 登入整合；username UI 文案與欄位型別仍留待後續前端階段。

## 實作內容

- `LoginRequest` 正式由 `email` 改為 `login`，契約為 required、string、最大 255 字元；密碼維持 required string。
- `AuthController` 只傳遞 validated `login` 與 password，維持薄 Controller。
- `AuthService` 先移除識別字前後空白，再依是否包含 `@` 分流：
  - Email：轉小寫後最多讀取兩筆相符資料；恰好一筆才可認證，case-variant 重複資料會 fail closed。
  - username：沿用 `User::normalizeUsername()`，以既有小寫儲存契約登入。
- 認證改以唯一解析出的 User ID 執行；不會由未排序的 Email 預查任選一個帳號。
- 不存在帳號與密碼錯誤都只回 `帳號或密碼錯誤`，不洩漏識別字是否存在。
- 停用帳號以 Email 或 username 登入都會登出並回既有停用訊息。
- 成功登入保留 Session regenerate 與 Authentication Audit；稽核寫入失敗仍會使 Session 失效。
- `UserResource` 新增 `username` 與 `must_change_password`，登入與 `/api/me` 共用相同契約。
- identifier + IP 與 account limiter 在可解析 User 時都使用 `uid:<id>` canonical identity；不存在或 Email 資料不唯一時使用 normalized login 的 SHA-256 key。
- Email／username alias、大小寫變體共用額度，既有 IP-wide、429 與 `Retry-After` 契約保留。
- `LoginRequest` 補齊繁體中文欄位錯誤訊息；API 文件同步停用帳號 422 與完整 `UserResource.role`。
