# v1.3 第 8 部分交接：發薪與 MoneyEntry 整合

日期：2026-07-14

## 完成範圍

`PLAN_v1.3.md` 第 8 部分已完成：

- `SalaryPeriodService::pay()` 只允許啟用中的 admin 對 confirmed 月份發薪；draft、paid、停用帳戶及非 admin 皆 fail-closed。
- 輸入契約為 `cash_account_id`、`payment_date` 與 `idempotency_key`。日期固定驗證 `YYYY-MM-DD`，批次 key 必填且最長 100 字元。
- 單一 transaction 先鎖 period，再鎖啟用中的資金帳戶與全部 settlement；每位 `net_pay > 0` 員工各建立一筆直接 approved 的正式薪資支出，零元員工不建立空 MoneyEntry。
- MoneyEntry 固定為 `expense`、`薪資 / 佣金`、`vehicle_id=null`、`source_type=salary_settlement`，對象為員工姓名，說明只含月份與薪資分類；不複製底薪、保險或獎金明細。
- 每筆支出成功後回填 settlement 的 `money_entry_id`；全部成功後才把 period 切為 paid，並保存帳戶、發薪日期、執行者與時間。任何中途例外都完整 rollback。
- 發薪完成即透過 approved MoneyEntry 反映資金帳戶餘額與 Dashboard monthly expense，不新增或保存 `current_balance`。
- 發薪 Audit Log 只記月份、操作與 paid 狀態，不保存員工薪資金額。

## Idempotency 與並發契約

- 批次 key 保存於 `salary_periods.idempotency_key`；相同月份、相同 key、相同帳戶與日期可 replay，且不重複寫 Audit Log 或 MoneyEntry。
- 相同 key 不同 payload 回 `idempotency_key` 422；同月份不同 key 的第二次發薪回 `status` 422；同一 key 不可跨月份重用。
- 單筆 MoneyEntry key 由批次 key 與 settlement id 經 SHA-256 衍生，不直接暴露可預測的固定 settlement key，避免一般收支預先占用命名空間。
- duplicate-key race 必須先讓原 transaction 完整 rollback，再開新 transaction 以 `lockForUpdate()` 讀取已提交 winner；不得在 MySQL REPEATABLE READ 舊快照內重讀。
- `SalarySettingsMysqlConcurrencyTest` 新增 fork/socket 真兩行程案例：輸家在寫入 period key 前停住，贏家於另一月份提交同 key；輸家恢復後撞 unique、完整 rollback，再於新 transaction 讀到 winner 並回 422，不殘留自己的 MoneyEntry 或 linkage。

## 已發薪歷史保護

新增 `2026_07_13_000012_protect_paid_salary_history` migration：

- paid period 不可再次 update 或 delete。
- paid period 不可新增、更新或刪除 settlement。
- paid settlement item 不可新增、更新或刪除。
- 既有 `salary_settlement` MoneyEntry 資料庫 trigger 持續阻擋 update／delete，一般 MoneyEntry CRUD 與 approval 流程也不開放異動。
- migration rollback 只移除本階段 triggers，不回寫歷史資料；forward migration 不做資料推定或 backfill。

歷史修正只能在後續月份建立有原因的手動調整，或由 admin 另建一般收支修正，不回改已發薪月份。
