## 3. API、URL 與權限基線

### Dashboard

- `GET /api/dashboard/summary` 由薄 Controller 呼叫 `DashboardService`，Service 以 transaction／REPEATABLE READ 提供同一資料快照。
- 現況只提供帳戶餘額、本月收支、淨流入、各車況與本月成交；v1.4 工作 KPI、毛利與 30 天趨勢尚未實作。
- `sales` 現況由 Controller 移除財務欄位；unknown role 同樣走非財務輸出。第 3 部分仍需依 v1.4 白名單 DTO／Resource 契約收斂。

### Vehicles

- 前端 `VehicleList` 目前只有 component state，沒有 URL 還原；API type 只接受單一 `status`。
- `IndexVehicleRequest` 只接受單一合法狀態；`VehicleService::listVehicles()` 只套單一狀態，尚未支援多狀態或 `is_preparation_completed`。
- 列表目前 eager load 收／賣車人，不載入封面；`VehicleResource` 回傳完整內部車輛欄位並依角色遮蔽價格。
- `Vehicle::photos()` 只包含已提交可見照片，且依 `sort_order` 排序；第 5 部分可在不改 schema 下加封面關聯／受限 eager load。
- `sales` 目前可看 `asking_price`、`floor_price`、`sold_price`。v1.4 Vehicle Card 規格要求 sales 卡片不呈現底價，這是列表呈現差異；後端既有詳細頁議價契約不可因此回歸。

### Money Entries

- 後端已支援搜尋、日期、方向、分類、帳戶、車輛、`approval_status` 與分頁。
- 前端目前全部使用 component state，且 `MoneyEntryList` 為取得車輛選項直接呼叫 `apiClient` 與 `per_page=100`；第 2 部分需將 URL 還原與 API 集中化一併處理。
- `salary_settlement`、sales 可見範圍與 cash account 遮蔽已由 Service／Policy／Resource 保護，v1.4 不可改變。

### Customers、Layout 與流程重用

- 建立車輛、建立客戶、一般收支建立頁與車輛詳情整備支出流程都已存在；Dashboard 快捷操作只能導向這些既有頁面。
- `AppLayout` 現有 Header 只包含 Mobile Sidebar 開關、使用者、主題與登出；v1.4 不在 Header 加快捷操作或通知。
- 本階段未新增 route、API、migration、schema、Workflow、狀態、角色或權限。

## 6. 第二輪 adversarial review 修正與驗證

第二輪 5 項 findings 逐項查證後均確認有實際風險，完成以下修正：

- CustomerSelect 查詢字串一變更就同步清空舊結果與 active option；loading 期間攔截方向鍵，避免對空清單操作。Enter 沒有 active option 時維持表單原生行為。
- active option 改變後以 `scrollIntoView({ block: 'nearest' })` 保持在 listbox 可視範圍內。
- 「搜尋中」改為一般狀態文字；空結果改用 `role="status"`，不再讓不可選訊息混入 `option` 集合。
- Sidebar 的 mobile／desktop 狀態改由 Sidebar 實際 computed `position` 同步，直接沿用 Tailwind `lg:` 已套用的 CSS 結果，不再由 JS 另寫一套 breakpoint。實測 Firefox 在非預設 20px 字級下，`matchMedia(rem)` 與 Tailwind 編譯 rule 的判斷並不一致，因此未採 review 建議的單純 rem 字串替換。
- focus trap 在 focus 已位於 Sidebar 外時會阻止該次 Tab，並將 focus 拉回第一個可操作元件。

Firefox headless 使用獨立 `/tmp` SQLite database、20 筆同姓客戶及非預設 20px 字級偏好實測：

- 1100px viewport 下 Sidebar 的 `inert`／`aria-hidden` 與 Tailwind 實際 mobile CSS 狀態一致。
- Sidebar 開啟後將 focus 程式化移到 `body`，按 Tab 會拉回關閉按鈕。
- 已載入「陳」姓 20 筆結果後立即改查「不存在」，loading 期間沒有舊 option、沒有 dangling `aria-activedescendant`，方向鍵加 Enter 不會選取舊客戶。
- 空結果只有 `role="status"`，沒有 `role="option"`。
- 對 20 筆結果連按 8 次 ArrowDown，active option 仍位於 listbox 可視範圍，容器 `scrollTop` 由 0 變為 242。

測試用 database、8001／5184 服務與 geckodriver 均已清除／停止，未接觸既有開發資料庫。

## 7. 第三輪 adversarial review 修正與驗證

第三輪 4 項 findings 均以最小範圍修正：

- loading 守衛不再攔截 Enter；舊結果仍由同步清空 `results`／`activeIndex` 根本防止誤選，Enter 可維持表單隱式送出。
- listbox 內固定保留空的 `role="status"` live region，查無結果時只更新文字；畫面上的相同提示以 `aria-hidden="true"` 避免重複播報。
- Sidebar computed position 判斷旁補上繁體中文維護註解，明確記錄它與 `lg:static`／mobile fixed position 的契約。
- resize 事件改由 `requestAnimationFrame` 合併同一影格內的重複呼叫，cleanup 同時移除 listener 並取消未執行 frame。

Firefox headless 使用獨立 `/tmp` SQLite database 實測：

- CustomerSelect 輸入新查詢後，在 `aria-busy=true` 的 debounce 期間按 Enter，表單 `submit` 事件觸發一次，輸入值未被既有客戶覆寫。
- 同一個預先掛載的 `role="status"` 元素從空字串更新為「查無相符客戶」，DOM 中仍只有一個 status region；可驗證 live region 的必要結構，但 headless browser 無法驗證 NVDA／VoiceOver 的實際語音輸出。

測試用 database、8001／5184 服務與 geckodriver 均已清除／停止，未接觸既有開發資料庫。
