# v1.3 第 6 部分交接：薪資資格與異常檢查

日期：2026-07-14

## 完成範圍

`PLAN_v1.3.md` 第 6 部分已完成：

- 新增無寫入副作用的 `SalaryEligibilityService`，集中處理月份候選車選取與資格檢查。
- `inspectPeriod()` 只選出台北月份內 `status=sold` 的車輛；非 sold 與其他月份不屬於本月候選。
- 每台候選車檢查收車人、賣車人、任意 pending MoneyEntry、approved 訂金／尾款扣退款、approved 購車付款、`legacy_unknown` 與 confirmed／paid 跨月份重複引用。
- approved 銷售淨收款必須達成交價；approved 購車付款必須與收購價完全相等，少付或多付都會阻擋。
- 異常車不會被靜默略過。結果保留 `stock_no`、問題碼、欄位、業務可讀訊息、修正標籤與前端可映射的 action；Service 不硬寫 SPA 路徑。
- `inspectVehicles()` 可對 transaction 內已鎖定的候選集合重驗；除月份選取外，仍會防禦性檢查 sold 狀態與成交月份。
- 草稿可使用 `inspectPeriod()` 顯示異常；確認流程只能在既有 transaction 內呼叫 `assertPeriodEligible()`。此入口會重新從 DB 選完整月份、鎖定候選車並以含車號與原因的 422 fail-closed，不接受舊草稿集合。
- 所有綁車 MoneyEntry 核准都會鎖定同一車輛列，維修等非銷售類支出核准也會與月份確認互斥。
- 訂金、尾款與退款分類集中至 `VehicleMoneyCategories`，由關帳、sales 可見範圍與薪資資格共用。
- 零毛利／虧損車只要其他資格完整仍是 eligible，並保留 approved-only 毛利。後續計算器會讓它留在明細與公司淨損益，但不推升賣車跨級台數。
- 本服務不依 category／status heuristic 自動修資料，也不推定歷史收車人或賣車人。

## 回傳契約

檢查結果包含：

```text
period_month
eligible_vehicles
vehicle_results[vehicle_id]
  ├─ vehicle / vehicle_id / stock_no
  ├─ eligible
  ├─ gross_profit
  └─ issues[]
anomalies[]
has_blocking_issues
```

`eligible_vehicles` 可直接交給 `SalaryCommissionCalculator`。`vehicle_results` 與 `anomalies` 應由第 7 部分 snapshot／Resource 使用，前端不得自行重做資格公式。

## 重要邊界

- `gross_profit <= 0` 不是資格異常。
- pending 的分類不限於銷售收款；任何綁定候選車的 pending MoneyEntry 都會阻擋。
- `legacy_unknown` 不論 approval status 都會阻擋，必須先完成人工來源確認。
- confirmed／paid 重複引用以 `salary_settlement_items.vehicle_id` 的 durable reference 判斷；draft 引用不視為歷史鎖定。
- 第 7 部分確認流程必須直接使用 `assertPeriodEligible()` 回傳的 `eligible_vehicles` 計算 snapshot；不可先從草稿取集合再呼叫 `inspectVehicles()`。
- 第 6 部分沒有新增 API 或前端頁面；月份草稿、Resource 與異常 UI 屬第 7／11 部分。
