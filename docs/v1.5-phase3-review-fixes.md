# ERPV2 v1.5 Phase 3 對立性審查修正

日期：2026-07-24

## 結論

Claude Review 的 F1～F5 均可重現或由實際契約確認，已採最小修正。F1 雖原列 `PLAN_v1.5.md` 第 4 部分，但雙識別登入已進入 `main`，不能把已知的 brute-force 額度加倍留在可部署基準，因此本輪依既有 PLAN 提前完成 canonical limiter 與全部 alias bypass tests。

## Finding 處理

### F1：Email／username alias 稀釋 limiter

- 有效。
- identifier + IP 與 account limiter 對已存在 User 統一使用 `uid:<id>`。
- 不存在或 Email 資料不唯一時使用 normalized login 的 SHA-256，避免 PII、超長 raw key 與不同不存在帳號互相污染。
- 補齊 Email→username、username→Email、大小寫、同 IP、rotating IP、成功登入清除與 `Retry-After` 回歸。

### F2：`LOWER(email)` 任選重複資料

- 有效。
- Email 查詢固定 `orderBy('id')->limit(2)`；只有恰好一筆才能取得 canonical User。
- 認證使用 canonical User ID，不再把任意 `value('email')` 當成登入依據。
- SQLite case-variant Email 並存時，兩組密碼都只回通用 422，沒有任何帳號會被誤認證。

### F3：前端仍送出 `email`

- 有效。
- 前端 Auth API 已最小改送 `{ login, password }`，Email 登入整合恢復。
- Login 頁 username 文案與 `type="text"` 仍依 PLAN 留待正式前端階段。

### F4：API 文件不一致

- 有效。
- `UserResource` 範例補上 `role`。
- 停用帳號登入狀態碼修正為實作與測試一致的 422。

### F5：Login Request 英文錯誤訊息

- 有效。
- `login` required／string／max 與 password required／string 均加入繁體中文訊息及精確回歸測試。

## 非 Finding 提醒

- Session fixation 專用測試仍依 PLAN §13.2 補強；本輪保留既有 regenerate 實作。
- Timing oracle 為既有 Laravel 認證行為，本輪不更換認證框架。
- `must_change_password` payload 保護依 PLAN §5 補 request 與 regression tests。
- MySQL／MariaDB Email case gated test 可在正式 Auth 測試階段補齊，本輪沒有 migration 或資料寫入變更。
