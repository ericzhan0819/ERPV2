## 1. 本階段完成內容

- 新增最小 Vitest runner，只作為 frontend dev dependency，不增加 runtime bundle dependency。
- `listFilters.test.ts` 鎖定 Vehicle／MoneyEntry Filter parse、serialize、工作區預設值、active predicate、direction／category／approval 白名單與正整數 ID。
- 抽出並實際供 Dashboard 使用的純展示契約：角色可見性、四個 Action Bar 連結、本月收支日期區間與成交月份車輛連結；測試 admin／manager／sales 與 fail-closed 行為。
- 抽出並實際供 Vehicle Card 使用的角色欄位契約；測試 admin／manager 只比 sales 多底價，所有角色都不顯示收購價或毛利。
- 對照 `DashboardSummaryResource`、`VehicleListResource`、frontend types 與 API modules，確認 Dashboard／Vehicle types 維持現行 JSON 契約。
- 靜態搜尋確認 API 呼叫仍集中在 `frontend/src/api`，頁面與元件沒有直接呼叫 endpoint。
- 查核 Dashboard、Vehicle List、MoneyEntry List 的 loading、error、空資料與 validation error 處理；未加入假 KPI、假趨勢或假車輛圖片。

## 2. 測試策略

本階段開始前 frontend 沒有 test runner。依 PLAN「可採最小方式、不得建立大型測試架構」的邊界，新增 Vitest 並只測純函式契約；沒有加入 jsdom、React Testing Library、Playwright 或新的 E2E 架構。這讓 URL、角色與欄位規則可快速回歸，同時避免為 v1.4 Presentation Layer 擴張測試基礎設施。
