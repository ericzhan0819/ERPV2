### API

Internal：

```text
GET    /api/vehicles/{vehicle}/photos
POST   /api/vehicles/{vehicle}/photos
PATCH  /api/vehicles/{vehicle}/photos/reorder
PATCH  /api/vehicles/{vehicle}/photos/{photo}/cover
DELETE /api/vehicles/{vehicle}/photos/{photo}
```

Public：

```text
GET /api/public/vehicles
GET /api/public/vehicles/{id}
```

Public API：

- 只公開 `status=listed`。
- 列表只載入封面照。
- 詳情頁才回傳完整公開相簿。
- 使用獨立白名單 Resource。
- 每 IP 每分鐘 60 次限流。
- 非上架與不存在車輛統一回 404。

## 5. 重要安全與一致性設計

### 5.1 Public Resource 白名單

公開 API 絕不可新增以下欄位：

- `purchase_price`
- `floor_price`
- `sold_price`
- VIN／車牌
- seller／buyer／customer 個資
- money entries
- cost／gross profit
- cash account
- internal notes
- approval status
- idempotency key

新增公開欄位時必須明確審查，不可改成共用內部 `VehicleResource`。

### 5.2 圖片權限

- `VehiclePolicy::viewPhotos()`：admin／manager／sales
- `VehiclePolicy::managePhotos()`：admin／manager
- sales 的寫入限制必須同時存在於 route policy 與前端 UI，不可只靠隱藏按鈕。

### 5.3 封面唯一性

- Service transaction 內序列化同車操作。
- DB 層 unique constraint／trigger 兜底。
- 刪除封面時自動選 `sort_order` 最小者。
- soft-deleted／hidden 殘留封面也必須清除，避免佔用 `cover_slot`。

### 5.4 上傳 idempotency

照片上傳沿用 replay-or-reject 原則，但因多檔案處理時間較長，使用 durable upload batch：

- 同 key／同 payload 可續傳或回放。
- 同 key／不同 payload 拒絕。
- claim token 防止舊 worker 寫回。
- processing lease 防止同批重複處理。
- stale batch 可由排程清理。
- batch 完成前照片不可見。

不要把這套機制簡化成單純「key 存在就成功」。

### 5.5 圖片處理全域鎖

`VehiclePhotoImageProcessor` 使用 database cache lock，強制同時間只處理一張大圖，避免 GD 原生記憶體疊加。

MySQL／MariaDB 連線必須保留：

```php
Mysql::ATTR_FOUND_ROWS => true
```

若移除，`DatabaseLock::refresh()` 可能在同一秒續約時誤判失敗，使真實 MySQL 上傳 100% 回報逾時。

## 6. 部署與環境必要步驟

首次部署或 public storage link 不存在時執行：

```bash
cd ~/ERPV2/backend
php artisan storage:link
```

驗證：

```bash
ls -l public/storage
```

應指向：

```text
/home/z/ERPV2/backend/storage/app/public
```

若缺少 link，圖片檔案即使已成功建立，`/storage/...` 仍會回 403／404。

目前 `.env` 使用：

```text
CACHE_STORE=database
```

未來若切換 cache store，需重新驗證 lock 的 atomic acquire／refresh／release 語意。
