# ERPV2 v1.4 企劃書：資訊架構與 UI／UX 改版

本文件是 `企劃書.md`、`企劃書_v1.1.md`、`企劃書_v1.2.md`、`企劃書_v1.3.md` 的延伸文件。

v1.1、v1.2、v1.3 均已封版。本文件不回開前述版本的功能範圍，也不藉由 UI 改版重新設計既有業務規則。

v1.4 是純 UI／UX 改版，核心工作是重新整理 ERPV2 的資訊架構、Dashboard 定位、車輛工作區、響應式體驗與可長期維護的 UX Design System。

---

## 1. 版本定位

ERPV2 是中古車行的營運管理系統。使用者每天面對的問題是：

```text
今天有哪些車需要處理？
```

而不是：

```text
今天要新增哪一筆會計資料？
```

因此 v1.4 將系統資訊架構明確分為：

```text
Dashboard = 總覽、KPI、趨勢、快速導流
各功能模組 = 搜尋、篩選、查看與實際操作的工作區
```

Dashboard 與模組不得合併。大量資料與工作操作必須留在各模組內。

---

## 2. v1.4 的硬性邊界

v1.4 不新增任何商業流程。

禁止：

- 修改 Database Schema
- 新增 migration 或資料表
- 修改既有 Business Logic
- 新增 Workflow 或狀態
- 新增角色或權限
- 新增通知功能或通知版位
- 為了快捷操作另建重複的表單、Modal、頁面或流程
- 以假資料或前端拼湊取代後端正式統計

允許的後端變更僅限 UI 必須的極少量 API 調整，例如：

- Dashboard 新增已定義的 KPI 與 30 天趨勢資料
- 車輛列表取得既有車輛照片中的封面縮圖
- 車輛列表支援既有欄位所需的可分享篩選條件
- 收支列表從 URL 還原既有審核篩選條件

上述 API 仍必須沿用既有 Service、Resource、Policy、approved-only、角色遮蔽與整數金額規則，不得改變正式計算口徑。

### 2.1 流程重用原則

若需求有兩種實作方式：

```text
A. 重用既有流程
B. 新增一套 UI 或 Workflow
```

一律採用 A。

除非企劃書明確要求新增流程，否則不得因 UI 改版產生新的操作路徑。

## 3. v1.4 目標

v1.4 必須完成：

- 重新整理整體資訊架構（Information Architecture）
- 分離 Dashboard 總覽與模組工作區的職責
- 建立一致的導流與 Filter 保留模式
- 把車輛列表由 Table 改為以圖片為主體的 Card Grid
- 改善 Desktop、Tablet 與 Mobile 操作體驗
- 修正 Safe Area、手機底部灰邊與底部留白
- 更新 `UI.md`，建立可長期維護的 UX Design System
- 保留既有 API、權限、資料來源與工作流程

本版不是只調整顏色、陰影或間距，也不是對所有頁面任意改版。

## 4. Dashboard 定位

Dashboard 只負責：

- 總覽
- KPI
- 趨勢
- 快速導流

Dashboard 不負責：

- 執行工作流程
- 顯示大量資料
- 展開車輛列表
- 顯示 Table
- 顯示車輛 Card 列表
- 在頁內展開 KPI 明細

Dashboard 永遠只是入口。使用者點擊 KPI 後，必須前往對應模組並帶入 Filter。

---

## 5. Dashboard 資訊架構

Dashboard 依下列順序排列：

```text
1. Dashboard 頁頭（Action Bar）
2. 工作概況
3. 經營概況
4. 趨勢分析
```

不得把 Action Bar 放進全域 Header。

### 5.1 Dashboard 頁頭（Action Bar）

v1.4 固定四個快捷功能，不增加其他項目：

| 快捷功能 | 可見角色 | 行為 |
|---|---|---|
| 建立收車 | admin、manager | 前往既有建立車輛頁 |
| 回報收支 | admin、manager、sales | 前往既有一般收支建立頁 |
| 建立客戶 | admin、manager、sales | 前往既有建立客戶頁 |
| 回報整備支出 | admin、manager、sales | 前往車輛模組並預先帶入 `status=preparing` |

「回報整備支出」不得新增選車 Modal 或獨立頁面。使用者在車輛模組搜尋並選擇車輛後，進入既有車輛詳情頁，沿用現有整備支出流程。

### 5.2 工作概況

工作概況使用可點擊的 Dashboard KPI Card。

所有角色皆可看到：

| KPI | 正式定義 | 導流結果 |
|---|---|---|
| 待整備 | `status=preparing` 且 `is_preparation_completed=false` | 車輛模組預載相同 Filter |
| 待上架 | `status=preparing` 且 `is_preparation_completed=true` | 車輛模組預載相同 Filter |
| 待交車 | `status=reserved` | 車輛模組預載相同 Filter |

admin 額外看到：

| KPI | 正式定義 | 導流結果 |
|---|---|---|
| 待審核收支 | `approval_status=pending` | 收支模組預載 pending Filter |

「待補獎金」不納入 v1.4，不得在 Dashboard 顯示薪資或獎金工作 KPI。

### 5.3 經營概況

Sales 只顯示：

- 在庫數

Manager／Admin 顯示：

- 在庫數
- 現金餘額
- 本月收入
- 本月支出
- 本月毛利
- 本月成交

正式定義：

```text
在庫數 = status IN (preparing, listed, reserved)
```

財務資料必須沿用既有規則：

- 現金餘額沿用目前 Dashboard 的 Cash Account 口徑
- 本月收入與本月支出只計 approved MoneyEntry
- 本月毛利沿用既有 approved-only 單車毛利計算方式
- 本月成交依 `sold_at` 與 `Asia/Taipei` 月份邊界計算
- 所有金額由後端計算，不得由前端自行加總

Sales 的原始 Dashboard JSON 不得包含任何財務 KPI 或財務趨勢欄位，不得回傳 `0`、空字串或假值代替遮蔽。

經營概況卡片若屬 KPI，點擊後只導向對應模組，不在 Dashboard 展開資料。導流後應保留對應月份、方向、狀態或帳戶類型等 Filter。

### 5.4 趨勢分析

v1.4 固定三項趨勢，不增加其他報表：

| 趨勢 | sales | manager | admin |
|---|---:|---:|---:|
| 近 30 天成交量 | 顯示 | 顯示 | 顯示 |
| 近 30 天毛利 | 不顯示 | 顯示 | 顯示 |
| 現金變化 | 不顯示 | 顯示 | 顯示 |

統計期間：

- 包含今天
- 依 `Asia/Taipei` 業務日期按日統計
- 共 30 個連續日資料點
- 無資料的日期仍回傳 `0` 或對應的每日期末餘額，確保圖表時間軸連續

近 30 天成交量：

- 依車輛 `sold_at` 所在日期計數

近 30 天毛利：

- 沿用既有單車毛利計算方式
- 只計 approved 收支
- 依成交車輛的 `sold_at` 日期歸入每日資料點
- 不另設計新的毛利公式

現金變化：

- 只統計 Cash Account
- 只計 approved 收支
- 沿用目前 Dashboard 現金餘額口徑
- 呈現每日結束時的現金餘額
- 不另設計新的財務邏輯

Dashboard 不放大型圖表。圖表必須維持可讀、低噪音，並在 Mobile 下不造成整頁水平捲動。

## 6. Dashboard 卡片互動

所有 Dashboard KPI Card 必須可點擊，但只能導流，不能展開明細。

基本流程：

```text
Dashboard KPI
→ 對應模組
→ URL 保存 Filter
→ 模組從 URL 還原 Filter
```

至少必須支援：

```text
待整備
→ /vehicles
→ status=preparing、is_preparation_completed=false

待上架
→ /vehicles
→ status=preparing、is_preparation_completed=true

待交車
→ /vehicles?status=reserved

待審核收支
→ /money-entries?approval=pending
```

URL query string 是 Filter 的正式分享與還原來源。進入模組後，使用者仍可搜尋、增加 Filter、修改 Filter 或清除 Filter。

既有 `status=preparing` 等單一狀態連結必須保持相容。若車輛工作區需要多狀態篩選，API 與前端可做最小擴充，但不得新增車輛狀態或改變狀態流程。

## 7. 車輛模組定位

車輛模組保留為獨立工作區，負責：

- 搜尋
- 篩選
- 分頁
- 查看車輛
- 編輯車輛
- 車輛收支
- 整備與上架
- 保留、收款與成交
- 其他既有車輛操作

Dashboard 不得複製上述功能。

## 8. 車輛列表改版

現有車輛 Table 改為 Card Grid：

```text
Desktop：Card Grid
Tablet：2 欄
Mobile：1 欄
```

每張 Vehicle Card 顯示：

- 車輛圖片
- 品牌
- 車型
- 年份
- 顏色
- 車牌
- 狀態
- 開價

Manager／Admin 額外顯示：

- 底價

列表卡片不顯示：

- 收購價
- 毛利

收購價與毛利只在既有權限允許的車輛詳細頁顯示。不得因列表隱藏而改變既有詳細頁權限，也不得只靠前端保護原本未授權的敏感資料。

車輛圖片是辨識主體，不是裝飾。列表只需取得既有照片中的封面縮圖，不載入整本相簿。無封面照片時使用一致的空狀態，不使用假車照片。

整張卡片應能清楚進入車輛詳細頁；卡片內狀態、價格與文字在 light／dark mode 均須可讀。

## 9. 車輛預設 Filter

車輛工作區初始狀態只顯示：

- `preparing`
- `listed`
- `reserved`

預設隱藏：

- `sold`
- `cancelled`

已完成或中止流程的車輛仍可透過 Filter 顯示，不得刪除、封存或改變資料。

Filter 必須支援：

- 搜尋既有車輛識別欄位
- 單一或多個狀態
- `is_preparation_completed`，供待整備／待上架導流使用
- 顯示 `sold`／`cancelled`
- 清除 Filter
- 分頁
- URL 還原與瀏覽器上一頁／下一頁

Dashboard 帶入的 Filter 優先於車輛工作區預設值。清除 Filter 後回到工作區的預設在庫狀態集合。

Mobile 的 Filter 使用 Drawer 呈現；Desktop／Tablet 保持可快速操作的篩選區。Drawer 只改變呈現方式，不新增不同的篩選邏輯。

## 13. API 與資料安全

Dashboard 的正式統計必須由後端 Dashboard Service 計算。前端只能格式化與繪製，不得自行拼湊：

- 現金餘額
- 本月收入
- 本月支出
- 本月毛利
- 本月成交
- 近 30 天毛利
- 現金每日餘額

所有財務彙總只計 `approval_status=approved`。

角色輸出：

- admin／manager 可取得本企劃指定的營運財務 KPI 與財務趨勢
- sales 只取得非財務工作 KPI、在庫數與近 30 天成交量
- 未授權財務欄位必須從原始 JSON 移除

車輛列表 Resource 必須維持既有敏感欄位規則。新增封面縮圖或 Filter 時，不得順便回傳完整相簿、收購價、毛利、資金帳戶或未授權收支資料。

## 16. 成功結果

```text
使用者進入 Dashboard
→ 立即看見今天需處理的工作、營運摘要與必要趨勢
→ 點擊 KPI 或快捷功能
→ 進入對應模組且 Filter 已預先套用
→ 在可承載大量資料的工作區搜尋、篩選並沿用既有流程完成工作
```

未來新功能必須遵守本文件的資訊架構：Dashboard 負責看與導流，模組負責大量資料與操作，不得因單一頁面需求破壞整體 UX 一致性。
