# ERPV2 v1.2 企劃書：車輛圖片與官網前置基礎

本文件是 `企劃書.md`、`企劃書_v1.1.md` 的延伸文件。v1.1 已完成 smoke 並打上 `v1.1-smoke-passed` tag，本文件不回開 v1.1 範圍。

v1.2 的目標不是直接完成完整官網，而是先補上官網必備的車輛圖片資料能力，讓 ERPV2 可以安全、穩定地管理車輛照片，並替未來官網 MVP 提供公開車輛資料基礎。

---

## 1. v1.2 產品定位

v1.2 定位為：

```text
車輛圖片管理 + 官網公開車輛資料前置版
```

核心目標：

```text
讓後台可以為每台車上傳、管理、排序、設定封面照片，並讓未來官網能安全讀取「已上架車輛」的公開照片與公開欄位。
```

v1.2 仍維持 ERPV2 的主產品邊界：

```text
ERPV2 是內部營運後台
官網是對外展示介面
後台管理公開資料來源
官網只讀公開資料，不碰內部成本、底價、成交價、客戶與收支
```

---

## 2. 為什麼圖片功能要獨立成 v1.2

官網 MVP 最重要的內容是車輛展示。中古車展示若沒有照片，價值很低。

車輛圖片牽涉：

```text
檔案上傳
檔案驗證
縮圖
排序
封面照
權限
公開 URL
未來本機 storage → Cloudflare R2 的搬移能力
公開 API 的資料遮蔽
```

這已經超出 v1.1 的角色、審核、客戶、入庫流程補強範圍，因此不應塞回 v1.1。

v1.2 只做圖片與公開展示基礎，不做完整 CMS 或正式官網。

---

## 3. v1.2 必須完成的功能

v1.2 包含五大項：

```text
1. 車輛照片資料模型
2. 車輛照片上傳 / 刪除 / 排序 / 封面照
3. 車輛詳情頁照片管理 UI
4. 公開車輛資料 API 基礎
5. 圖片儲存架構預留本機 storage → Cloudflare R2 遷移能力
```

## 4. 車輛照片資料模型

新增資料表：

```text
vehicle_photos
```

建議欄位：

| 欄位 | 說明 |
|---|---|
| `id` | 主鍵 |
| `vehicle_id` | 關聯車輛 |
| `disk` | 儲存 disk，例如 `public`、未來 `r2` |
| `path` | 原始或主要顯示圖路徑 |
| `thumbnail_path` | 縮圖路徑 |
| `original_filename` | 原始檔名，只供內部參考 |
| `mime_type` | MIME 類型 |
| `size` | 檔案大小 bytes |
| `width` | 圖片寬度 |
| `height` | 圖片高度 |
| `sort_order` | 排序 |
| `is_cover` | 是否封面照 |
| `uploaded_by` | 上傳者 user id |
| `created_at` / `updated_at` | 時間戳 |

重要原則：

```text
資料庫只記 disk + path，不記死本機絕對路徑。
```

這樣未來從本機 storage 搬到 Cloudflare R2 時，不需要重寫整個圖片系統。

## 5. 圖片儲存策略

### 5.1 v1.2 預設：Laravel 本機 public storage

第一版先用 Laravel 內建 storage：

```text
storage/app/public/vehicles/{vehicle_id}/...
```

並使用：

```bash
php artisan storage:link
```

這樣開發最快，也不需要先處理雲端 bucket、金鑰、public policy 或 CDN。

### 5.2 架構必須預留 Cloudflare R2

雖然 v1.2 不直接導入 R2，但設計上必須保留：

```text
disk: public / r2
path: vehicles/{vehicle_id}/{uuid}.webp
thumbnail_path: vehicles/{vehicle_id}/{uuid}_thumb.webp
```

禁止在資料庫寫入：

```text
/home/z/ERPV2/backend/storage/app/public/...
```

也不要把圖片 URL 寫死成開發機 IP。

### 5.3 圖片尺寸策略

不要直接在列表或官網載入手機原圖。

建議最少產生：

```text
thumbnail：列表 / 管理頁小圖
display：詳情頁與官網展示圖
original：可選，內部保留原始上傳檔
```

v1.2 最小必須做：

```text
path：主要顯示圖（重新編碼為 webp，移除 EXIF）
thumbnail_path：縮圖（重新編碼為 webp）
```

**最低處理要求**：
- 重新編碼為 webp（移除上傳原始檔的 EXIF / GPS 隱私資訊）
- 產生縮圖（減少網路頻寬，供管理頁小圖與列表使用）
- 原始上傳檔可選保留，但公開 API 與詳情頁不可直接使用原始檔

若需引入圖片處理套件（例如 `intervention/image`），必須在實作報告中說明理由。

## 6. 上傳限制與安全規則

### 6.1 檔案格式

允許：

```text
jpg
jpeg
png
webp
```

不允許：

```text
svg
heic
pdf
影片
壓縮檔
可執行檔
```

SVG 不開放，避免 script / XSS 風險。

### 6.2 檔案大小

建議限制：

```text
單張最大 8MB
每台車最多 60 張
一次上傳最多 20 張
```

實作時可依前端體驗調整，但必須有明確上限，避免誤傳大量照片打爆主機。

### 6.3 檔名

公開儲存檔名必須使用系統產生的隨機檔名，例如 UUID。

不可直接使用使用者上傳的原始檔名作為 storage path。

### 6.4 EXIF / GPS 隱私

公開展示用圖片應避免保留手機拍攝的 EXIF / GPS 定位資訊。

最小策略：

```text
縮圖與主要展示圖經重新編碼產生，不保留 EXIF。
```

若保留原圖，原圖不得直接作為官網公開圖片使用。

## 7. 權限設計

| 功能 | admin | manager | sales | public |
|---|---:|---:|---:|---:|
| 查看後台車輛照片 | 是 | 是 | 是 | 否 |
| 上傳車輛照片 | 是 | 是 | 否 | 否 |
| 刪除車輛照片 | 是 | 是 | 否 | 否 |
| 設定封面照 | 是 | 是 | 否 | 否 |
| 調整照片排序 | 是 | 是 | 否 | 否 |
| 讀取公開已上架車輛照片 | 不需登入 | 不需登入 | 不需登入 | 是 |

sales 可看照片，是因為照片屬於銷售作業需要。

sales 暫不開放上傳、刪除、排序，避免初期資料治理混亂。若試跑後確認 sales 需要直接補照片，可在 v1.2.x 或 v1.3 再開放「上傳待審」模式。

## 8. 後台照片管理流程

### 8.1 車輛詳情頁

車輛詳情頁新增區塊：

```text
車輛照片
```

顯示：

```text
封面照
照片縮圖 grid
照片數量
上傳者 / 上傳時間（可選）
```

admin / manager 可操作：

```text
上傳多張照片
刪除照片
設定封面照
調整排序
```

sales 只能查看。

### 8.2 封面照規則

每台車最多只能有一張 `is_cover=true`。

規則：

```text
第一張上傳照片自動成為封面
手動設定封面時，同一車輛其他照片自動取消封面
刪除封面照時，若仍有其他照片，系統自動選 sort_order 最小者為新封面
```

### 8.3 排序規則

照片排序使用 `sort_order`。

拖拉排序或上移 / 下移皆可，實作時以穩定、簡單為優先。

排序更新必須限制在同一台車內，不可跨車輛改到其他車照片。

## 9. 內部 API 範圍

建議新增內部 API：

```text
GET    /api/vehicles/{vehicle}/photos
POST   /api/vehicles/{vehicle}/photos
PATCH  /api/vehicles/{vehicle}/photos/reorder
PATCH  /api/vehicles/{vehicle}/photos/{photo}/cover
DELETE /api/vehicles/{vehicle}/photos/{photo}
```

### 9.1 Resource 回傳

`VehiclePhotoResource` 建議回傳：

```json
{
  "id": 1,
  "vehicle_id": 12,
  "url": "http://localhost/storage/vehicles/12/photo.webp",
  "thumbnail_url": "http://localhost/storage/vehicles/12/photo_thumb.webp",
  "is_cover": true,
  "sort_order": 1,
  "width": 1200,
  "height": 800,
  "size": 245000,
  "created_at": "2026-07-08T00:00:00+08:00"
}
```

不要在 API 回傳本機絕對路徑。

## 10. 公開車輛資料 API 基礎

v1.2 可新增公開唯讀 API，供未來官網 MVP 使用。

建議端點：

```text
GET /api/public/vehicles
GET /api/public/vehicles/{id}
```

公開條件：

```text
status = listed
is_active = true（若車輛有此欄位）
```

如果目前沒有 `is_public` 欄位，v1.2 不一定要新增；第一版可用 `status=listed` 作為公開依據。若實務上需要「上架但不公開」，再新增 `is_public`。

### 10.1 公開 API 可回傳

```text
stock_no
brand
model
year
mileage
color
fuel_type
transmission
displacement
asking_price
public_description（若 v1.2 有新增）
cover_photo
photos
created_at / listed_at
```

### 10.2 公開 API 絕對不可回傳

```text
purchase_price
floor_price
sold_price
buyer / seller / customer_id
buyer_name / buyer_phone
seller_name / seller_phone
money_entries
cash_account
cost summary
gross_profit
internal notes
approval_status
idempotency_key
任何客戶個資
```

公開 API 必須有獨立 Resource，不可直接拿內部 `VehicleResource` 對外使用。

## 12. 測試與驗收

v1.2 必須有自動化測試覆蓋：

```text
admin / manager 可上傳照片
sales 不可上傳 / 刪除 / 排序 / 設封面
sales 可查看照片
未登入不可讀內部照片 API
檔案格式驗證
檔案大小驗證
照片數量上限
第一張照片自動成為封面
設定封面時同車其他照片自動取消封面
刪除封面後自動補封面
排序只能影響同一車輛
刪除照片時資料列與 storage 檔案一致處理
公開 API 只回傳 listed 車輛
公開 API 不回傳底價、成交價、收購價、客戶、收支、毛利、資金帳戶
```

Frontend 必須至少驗證：

```text
車輛詳情頁可看到照片區塊
admin / manager 看到管理按鈕
sales 不看到管理按鈕
無照片時有清楚空狀態
上傳失敗有清楚錯誤提示
```
