# ERPV2 v1.5 企劃書：帳號自助管理與系統識別集中化

本文件是 `企劃書.md`、`企劃書_v1.1.md`、`企劃書_v1.2.md`、`企劃書_v1.3.md`、`企劃書_v1.4.md` 的延伸文件。

v1.1～v1.4 均已封板。本版本不回開既有車輛、收支、薪資、Dashboard 或 UI／UX 改版範圍，只補齊目前尚未完成的登入便利性、首次登入安全流程、個人帳號維護，以及 ERPV2 系統名稱集中設定。

---

## 1. 版本定位

v1.5 是一個小型但完整的帳號體驗補強版本，解決兩個實際問題：

1. 員工目前只能使用 Email 登入，手機輸入過長且不方便。
2. ERPV2 內仍有「中古車行系統」「中古車行內部營運系統」等硬編碼文字，公司名稱確定後若直接逐頁修改，容易漏改。

v1.5 正式定位：

```text
管理員建立員工帳號與預設密碼
→ 員工第一次登入後必須修改密碼
→ 員工可自行設定短帳號名稱
→ 後續可用帳號名稱或 Email 登入
→ 員工可自行修改顯示名稱與密碼
→ ERPV2 的系統名稱集中於前端 config 管理
```

本版不是完整 Settings Module，也不是 HR、SSO 或企業身分管理系統。

## 2. 已確認的現況

目前程式已確認：

- `users` 具備 `name`、`email`、`password`、`role`、`is_active` 與員工資料欄位。
- Email 已有 unique constraint。
- 登入只接受 `email + password`。
- 登入已有帳號／IP Rate Limiter、停用帳號保護、Session regenerate 與登入稽核。
- admin 已可建立員工、編輯基本資料、修改角色、啟用／停用、重設密碼與刪除未被引用的帳號。
- Header 已顯示登入者名稱、Theme Toggle 與登出。
- Theme 已透過 localStorage 保存，不需要納入 v1.5 個人設定。
- `/api/me` 已回傳目前登入者資料。
- `User` model 已受 Audit Observer 監控，密碼欄位已有隱藏與稽核遮蔽基礎。
- `frontend/index.html`、登入頁與 Sidebar 仍存在硬編碼系統名稱。

目前尚無：

- `users.username`
- `users.must_change_password`
- username／Email 雙識別登入
- 首次登入強制修改密碼
- 員工自助「我的帳號」頁
- ERPV2 統一文字型 app config

## 3. v1.5 目標

v1.5 必須完成：

- 管理員建立新員工時，仍只需填名稱、Email、預設密碼、角色與既有員工欄位。
- 新建立的員工帳號必須標記為首次登入需改密碼。
- admin 重設任何員工密碼後，該員工下次操作前必須再次修改密碼。
- 員工登入後可自行修改顯示名稱。
- 員工可新增或修改短帳號名稱 `username`。
- Email 維持管理員管理，員工只能查看，不能自行修改。
- 員工可修改自己的密碼，並必須輸入目前密碼與新密碼確認。
- 登入畫面接受「帳號名稱或 Email」。
- Username 與 Email 登入必須共用既有安全限制，不能藉由切換識別方式繞過 Rate Limiter。
- `must_change_password=true` 時，前後端都必須阻擋一般營運功能，不能只靠前端導頁。
- ERPV2 系統名稱與瀏覽器標題集中於前端 config。
- 既有 Theme Toggle、角色權限、使用者管理與 v1.1～v1.4 功能不得回歸。

## 4. 正式帳號流程

### 4.1 管理員建立員工

管理員沿用既有使用者管理頁，填寫：

```text
名稱
Email
預設密碼
角色
既有員工資料欄位
```

建立結果：

```text
username = null
must_change_password = true
```

管理員不需要替員工先決定 username。員工可先使用 Email 與預設密碼完成第一次登入。

### 4.2 員工第一次登入

```text
Email + 預設密碼
→ 登入成功
→ 系統判定 must_change_password=true
→ 導向強制修改密碼頁
→ 一般營運 API 與頁面暫時不可使用
→ 修改密碼成功
→ must_change_password=false
→ 進入 Dashboard
```

強制修改密碼流程只強制更換密碼，不強制員工立即設定 username。員工可於同一個「我的帳號」頁設定 username，也可先繼續用 Email 登入。

### 4.3 後續登入

員工可使用：

```text
username + password
```

或：

```text
email + password
```

兩種登入方式對應同一個 User、同一組密碼、同一個啟用狀態與同一組 Rate Limiter。

### 4.4 管理員重設密碼

管理員重設密碼後：

```text
password = 新預設密碼
must_change_password = true
```

若員工當時已有登入 Session，下一個受保護 API 請求也必須被強制改密碼 Middleware 阻擋，不能讓既有 Session 長期繼續使用。

## 5. User Schema 決策

新增兩個欄位：

```text
username               nullable string, unique
must_change_password   boolean
```

### 5.1 `username`

用途：提供短登入帳號。

正式規則：

- nullable：管理員建立帳號時不必填。
- unique：全系統不可重複。
- 長度 3～30 字元。
- 只允許英文字母、數字、句點、底線與連字號。
- 不允許 `@`，避免與 Email 識別混淆。
- 儲存前 trim 並轉小寫。
- 登入時不區分大小寫。
- 空字串正規化為 `null`。

允許範例：

```text
eric
sales01
wang.xm
manager_1
```

不允許範例：

```text
王小明
eric@example.com
sales 01
```

### 5.2 `must_change_password`

用途：表示使用者在進入一般營運功能前是否必須更換密碼。

正式規則：

- Database default 為 `false`，避免 migration 將既有使用者全部鎖住。
- admin 新建使用者時由 Service 明確設為 `true`。
- admin 重設密碼時設為 `true`。
- 使用者成功修改自己的密碼後設為 `false`。
- 既有使用者 migration 不做推測式回填。
- Seeder／Factory 必須可明確建立已完成或待完成首次改密碼的狀態。

## 6. 登入識別契約

登入 Request 由：

```json
{
  "email": "admin@example.com",
  "password": "..."
}
```

調整為：

```json
{
  "login": "eric 或 admin@example.com",
  "password": "..."
}
```

### 6.1 識別規則

- `login` 先 trim。
- 含 `@` 時以 Email 查詢。
- 不含 `@` 時以 normalized username 查詢。
- Email 與 username 都採不區分大小寫的登入行為。
- 不論帳號不存在、密碼錯誤或識別方式錯誤，對外都維持通用錯誤訊息：`帳號或密碼錯誤`。
- 停用帳號仍回既有停用訊息。
- 登入成功仍須 regenerate Session 並留下 Authentication Audit Log。

### 6.2 Rate Limiter

既有登入限制不得因雙識別登入而退化。

正式要求：

- 保留 IP-wide limiter。
- 保留 identifier + IP limiter。
- 同一個實際 User 的 username 與 Email 必須共用 canonical account limiter。
- 攻擊者不能先用 username 打滿一半，再改用 Email 取得另一組完整額度。
- 不存在的 identifier 仍需有穩定 limiter key，避免無限嘗試。
- 成功登入只清除該次 identifier／IP 與 canonical account limiter，不清除 IP-wide 失敗保護以外的不相關帳號。
- 既有 429 與 `Retry-After` 契約保留。

## 7. 首次登入強制改密碼

### 7.1 後端必須強制

只做前端 redirect 不足以構成安全保護。v1.5 必須新增 authenticated middleware 或等價後端 gate。

當：

```text
request user.must_change_password = true
```

一般營運 API 回傳 machine-readable 錯誤，例如：

```json
{
  "message": "請先修改密碼",
  "code": "PASSWORD_CHANGE_REQUIRED"
}
```

HTTP status 固定採 `409 Conflict`，代表登入有效，但目前帳號狀態與一般操作衝突。

### 7.2 強制狀態允許的 API

待改密碼使用者只允許：

- `GET /api/me`
- `POST /api/logout`
- 更新自己的基本資料
- 更新自己的密碼

不得存取：

- Dashboard
- 車輛
- 客戶
- 收支
- 資金帳戶
- 使用者管理
- 薪資
- 稽核紀錄
- 其他一般營運 API

### 7.3 前端導流

- 登入 API 或 `/api/me` 回傳 `must_change_password=true` 時，導向強制修改密碼頁。
- 強制修改密碼頁不顯示一般 Sidebar 與營運導覽。
- 保留 Theme Toggle 與登出能力。
- 修改成功後更新 Auth Context，再導向 Dashboard。
- 直接輸入其他前端 URL 時仍會被 Protected Route 導回強制頁。
- 即使前端狀態失步，後端 409 也必須觸發相同導流。

## 8. 我的帳號

所有已登入且啟用的角色均可使用：

```text
admin
manager
sales
```

### 8.1 個人資料

顯示：

- 顯示名稱
- 帳號名稱
- Email
- 角色

可修改：

- 顯示名稱
- username

唯讀：

- Email
- 角色
- 啟用狀態

Email 仍由 admin 在使用者管理頁維護，不開放員工自行修改。

### 8.2 修改密碼

欄位：

```text
目前密碼
新密碼
確認新密碼
```

規則：

- 必須驗證目前密碼。
- 新密碼沿用現有至少 8 字元規則。
- 新密碼與確認密碼必須一致。
- 新密碼不可在 API response、Audit Log、前端 log 或錯誤資訊中出現。
- 修改成功後保留目前登入 Session，重新 regenerate Session ID。
- 修改成功後清除 `must_change_password`。

v1.5 不做裝置管理、其他 Session 列表或一鍵登出所有裝置。

### 8.3 入口

優先沿用 Header 現有使用者名稱區域提供「我的帳號」入口，不新增 Sidebar 主模組項目，避免把個人帳號維護誤認為營運功能。

Theme Toggle 保留在 Header，不搬入「我的帳號」。

```text

```

Request：

```json
{
  "login": "eric",
  "password": "password"
}
```

Response `UserResource` 新增：

```text

```

### 11.2 Current User Profile

```text
PATCH /api/me/profile
```

Request：

```json
{
  "name": "王小明",
  "username": "xiaoming"

```

只允許目前登入者修改自己的 `name` 與 `username`。

### 11.3 Current User Password

```text

```

Request：

```json
{
  "current_password": "舊密碼",
  "password": "新密碼",
  "password_confirmation": "新密碼"
}
```

成功後：

```text
must_change_password=false
```

### 11.4 Admin User API

既有：

```text
POST /api/users
POST /api/users/{user}/reset-password
```

行為補強：

```text
create user          → must_change_password=true
reset user password  → must_change_password=true
```

## 12. 權限與安全邊界

- 任何 authenticated user 只能修改自己的 `name`、`username` 與 password。
- 使用者不能透過 self API 修改 Email、role、is_admin、is_active、phone、job_title、hire_date、notes 或薪資設定。
- Admin 使用者管理權限維持既有 admin-only。
- `username` 與 `must_change_password` 必須由後端驗證，不只靠前端表單。
- `must_change_password` gate 必須 fail-closed。
- 未知角色若已通過登入，也只能修改自己的安全自助欄位，不能取得額外營運權限。
- 密碼欄位、目前密碼與 confirmation 永遠不得進入 Resource、Audit Log 或一般 log。
- Username／Email 登入不得洩漏帳號是否存在。
- 停用帳號仍不得登入。

## 13. Migration 與既有資料

Migration 必須：

- 新增 nullable unique `username`。
- 新增 `must_change_password` boolean default false。
- 不回填 username。
- 不從 Email local-part、姓名或其他欄位推測 username。
- 不把既有帳號全部設為待改密碼。
- down migration 清楚移除 unique index 與兩欄位。

既有資料升級後：

```text
username = null
must_change_password = false
```

這確保目前本機管理員與測試帳號不會因 migration 立即被鎖在強制頁。

---

## 14. Audit 與可追溯性

應留下稽核：

- admin 建立使用者。
- admin 重設密碼造成 `must_change_password` 變更。
- 使用者修改自己的 name。
- 使用者新增或修改 username。
- 使用者完成密碼修改造成 `must_change_password=false`。

稽核不得包含：

- password
- password hash
- current_password
- password_confirmation

Authentication audit 仍須記錄登入／登出；可辨識 actor 的 username 可由 User 關聯查詢，但不需要把登入密碼或原始 login input 寫入 Audit Log。
