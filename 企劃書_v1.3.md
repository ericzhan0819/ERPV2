# ERPV2 v1.3 企劃書：薪資結算

本文件是 `企劃書.md`、`企劃書_v1.1.md`、`企劃書_v1.2.md` 的延伸文件。

v1.1 已以 `v1.1-smoke-passed` 封版，v1.2 已以 `v1.2-smoke-passed` 封版。本文件不回開前述版本範圍。

v1.3 的目標不是建立完整 HR、打卡或法定薪資申報系統，而是把中古車行目前最容易人工算錯、最需要公式化的薪資與獎金結算流程納入 ERPV2：

```text
車輛成交
→ 系統依已核准收支計算單車毛利
→ 套用公司保留與獎金分配規則
→ 依每位賣車人當月成交台數套用跨級階梯
→ 彙總底薪、收車獎金、賣車獎金、加扣項、勞保與健保扣款
→ 老闆確認
→ 發薪並自動建立薪資支出 Money Entry
```

---

## 1. 產品定位

v1.3 定位為：

```text
中古車行薪資結算
```

核心目標：

```text
讓老闆不用再拿計算機或 Excel 手算每位員工當月應發薪資，
而是由系統以固定公式、正式成交資料與已核准收支自動試算，
經老闆確認後再把實際發薪金額記入資金支出。
```

v1.3 不是：

```text
完整人資系統
完整 Payroll 法規引擎
打卡系統
排班系統
請假系統
勞健保級距自動申報系統
所得稅扣繳申報系統
```

---

## 2. 為什麼薪資結算要排在官網之前

目前 ERPV2 已完成：

```text
收車
→ 建檔
→ 整備支出
→ 上架
→ 收訂金
→ 收尾款
→ 成交結案
```

但成交後仍需要人工整理：

```text
每位員工賣了幾台
每台車毛利多少
哪一台是誰收的
哪一台是誰賣的
跨級後全部車輛應套用多少比例
底薪與勞健保扣款多少
最後應發多少
發薪後有沒有記入支出
```

這些工作若留在計算機或 Excel，最容易發生：

- 漏算車輛
- 重複算獎金
- 把尚未收足款的車算入
- 收車人或賣車人歸屬錯誤
- 跨級後漏補前面車輛差額
- 調整規則後回頭改變歷史結算
- 已發薪卻沒有記入資金帳戶
- 同一月份重複發薪

v1.3 要用可追溯、可鎖定、可重算草稿但不可竄改已發薪結果的方式解決。

## 3. 核心獎金規則

### 3.1 單車毛利

v1.3 沿用系統正式的 approved-only 原則：

```text
單車毛利
= 該車 approved 收入合計
- 該車 approved 支出合計
```

不得計入：

- pending 收入或支出
- rejected 收入或支出
- 未綁定該車的一般營運收支

為避免毛利因資料未完成而被高估，薪資結算前需檢查：

- 車輛狀態必須為 `sold`
- `sold_at` 必須依 `Asia/Taipei` 業務日期落在結算月份
- 訂金與尾款已依既有成交結案規則核准並收足
- 不可仍有該車 pending 的金流資料
- 已核准購車付款應與 `purchase_price` 一致；若不一致，列為待處理異常，不可直接確認結算
- 收車人與賣車人必須明確指定

不得用 `created_by`、`updated_by`、最後操作人或其他 heuristic 推定收車人／賣車人。

### 3.2 公司營運保留

預設：

```text
公司營運保留比例 = 單車毛利的 40%
```

公式：

```text
公司營運保留
= 單車毛利 × 40%
```

這部分不受賣車跨級階梯影響，永遠先保留給公司作為管銷與營運資金。

### 3.3 可分配獎金池

預設：

```text
可分配獎金池比例 = 單車毛利的 60%
```

公式：

```text
可分配獎金池
= 單車毛利 - 公司營運保留
```

實作上不應另外獨立計算 60% 後再與 40% 四捨五入，以免一元誤差。應先計算公司營運保留，再以毛利減除得到分配池。

### 3.4 收車獎金

預設：

```text
收車獎金 = 可分配獎金池的 20%
```

收車獎金不受賣車台數階梯影響。

每台車必須有明確 `purchase_agent_id`。同一人可以同時是收車人與賣車人，並同時取得兩種獎金。

### 3.5 賣車獎金：跨級階梯

賣車獎金採「跨級後整月追溯套用」制度，不採逐台邊際級距。

符合資格成交台數只計入 `gross_profit > 0` 的車輛。零毛利或虧損車仍保留在月份明細與公司淨損益，但不增加賣車跨級台數，避免虧損成交推升其他車輛的獎金比例。

預設級距：

| 每位賣車人當月符合資格的成交台數 | 整月賣車獎金比例 |
|---:|---:|
| 0 台 | 0% |
| 1～2 台 | 分配池 20% |
| 3～4 台 | 分配池 30% |
| 5 台以上 | 分配池 50% |

例如某位業務當月前四台暫按 30% 試算，第五台符合資格後：

```text
該業務當月五台車的賣車獎金比例全部改為 50%
```

系統畫面必須清楚顯示：

- 當月符合資格成交台數
- 目前適用級距
- 距離下一級還差幾台
- 目前預估賣車獎金
- 跨級後增加的預估金額

### 3.6 公司剩餘分配額

公式：

```text
公司剩餘分配額
= 可分配獎金池
- 收車獎金
- 賣車獎金
```

公司最終保留：

```text
公司最終取得
= 公司營運保留
+ 公司剩餘分配額
```

最高級距下：

```text
分配池 100%
- 收車獎金 20%
- 賣車獎金 50%
= 公司剩餘分配額 30%
```

因此賣車人跨級不會吃掉公司先保留的 40% 毛利。

### 3.7 公式範例

單車毛利：`100,000`

```text
公司營運保留：100,000 × 40% = 40,000
可分配獎金池：100,000 - 40,000 = 60,000
收車獎金：60,000 × 20% = 12,000
賣車獎金（最高級距 50%）：60,000 × 50% = 30,000
公司剩餘分配額：60,000 - 12,000 - 30,000 = 18,000
公司最終取得：40,000 + 18,000 = 58,000
```

必須維持下列不變量：

```text
公司營運保留
+ 收車獎金
+ 賣車獎金
+ 公司剩餘分配額
= 單車毛利
```

所有金額使用整數新台幣，不使用 float。百分比使用 basis points 或等價整數設計。各獎金項目向下取整，所有餘數歸入公司剩餘分配額，確保總額不遺失、不超配。

### 3.8 虧損與零毛利

```text
單車毛利 <= 0
→ 公司營運保留 = 0
→ 可分配獎金池 = 0
→ 收車獎金 = 0
→ 賣車獎金 = 0
```

虧損金額仍需在結算明細顯示，但不得產生負獎金或自動倒扣員工薪資。若要追究特殊責任，使用 admin 手動扣款項目並填寫原因，不把責任判斷寫進自動公式。

## 4. 獎金方案

v1.3 不把 40%、20%、20%／30%／50% 散落寫死在 Controller 或前端。

需建立可版本化的獎金方案：

```text
Commission Plan / 獎金方案
```

最低欄位：

- 名稱
- 生效日
- 公司營運保留比例
- 收車獎金比例
- 賣車跨級階梯
- 是否啟用
- 建立者
- 建立時間

初始方案：

```text
名稱：2026 標準薪資方案
公司營運保留：40%
收車獎金：分配池 20%
賣車級距：
- 1～2 台：20%
- 3～4 台：30%
- 5 台以上：50%
```

規則：

- 一個月份只能套用一個方案
- 已被薪資結算引用的方案不得直接修改比例或級距
- 規則改變時建立新方案與新生效日
- 已確認或已發薪月份永遠保留當時的規則快照
- 不允許修改新方案後回頭改變歷史薪資

## 5. 員工薪資設定

每位需要結算薪資的使用者需有薪資設定：

- 月底薪
- 固定津貼（可為 0）
- 員工負擔勞保扣款
- 員工負擔健保扣款
- 是否參與收車／賣車獎金
- 生效狀態

第一版不自行實作官方勞健保級距與費率公式。

原因：

- 投保級距不一定等於底薪
- 費率與級距可能調整
- 健保可能有眷屬等差異
- 本系統不是法定申報系統

v1.3 的做法是由 admin 為每位員工設定當期固定扣款金額，系統在每月結算時複製快照。

即使之後修改員工薪資設定，也不得改變已確認月份。

---

## 6. 薪資公式

每位員工每月：

```text
應發合計
= 底薪
+ 固定津貼
+ 收車獎金合計
+ 賣車獎金合計
+ 其他加給
```

```text
扣款合計
= 勞保扣款
+ 健保扣款
+ 其他扣款
```

```text
實發薪資
= 應發合計
- 扣款合計
```

其他加給／其他扣款：

- 只有 admin 可新增
- 必須填寫原因
- 金額必須為正整數，方向由項目類型決定
- 不允許直接覆寫自動算出的收車或賣車獎金

實發薪資不得小於 0；若公式結果小於 0，阻擋確認並要求人工處理。

## 7. 收車人與賣車人歸屬

目前 `vehicles` 只有 `created_by`／`updated_by`，不足以代表實際收車人與賣車人。

v1.3 需新增：

```text
purchase_agent_id  收車人
sales_agent_id     賣車人
```

規則：

- 新增車輛時由 admin／manager 明確指定收車人
- 執行保留／成交銷售流程時明確指定賣車人
- sales 自己執行銷售流程時，可預設為目前登入者，但後端仍需保存正式欄位
- admin／manager 代為登記時必須選擇實際賣車人
- 已售出但尚未結算的歷史車輛，需由 admin 人工補指定
- 不得用建立者、更新者、收款建立者或最後操作者自動回填歷史歸屬
- 薪資月份確認後，不得更改該月份已引用車輛的收車人／賣車人

若缺少收車人或賣車人，該車列入「待補資料」，薪資月份不可確認。

## 8. 薪資結算生命週期

### 8.1 預估

系統可隨時依目前資料計算本月預估：

- 每位員工目前符合資格成交台數
- 目前跨級比例
- 收車獎金
- 賣車獎金
- 底薪與固定扣款
- 預估實發薪資
- 全公司本月預估薪資支出

預估不代表已確認，不建立 Money Entry。

### 8.2 建立月份草稿

admin 建立指定月份薪資結算草稿。

系統需：

- 選定該月份有效獎金方案
- 讀取符合資格成交車輛
- 計算每位賣車人當月成交台數與跨級比例
- 為每台車建立可追溯的收車／賣車獎金明細
- 複製員工底薪、津貼、勞保、健保設定快照
- 顯示所有阻擋異常

草稿狀態可重新計算，但重新計算前需明確提示會重建自動項目；手動加扣項不得被靜默刪除。

### 8.3 確認結算

只有 admin 可確認。

確認前必須：

- 所有車輛歸屬完整
- 無 pending 車輛收支
- 購車付款與收購價一致
- 公式不變量成立
- 每位員工實發薪資不小於 0
- 沒有重複月份或重複車輛獎金

確認後：

- 結算與明細鎖定
- 不可重新計算
- 不可修改獎金方案快照
- 不可修改底薪與扣款快照
- 不可直接刪除

### 8.4 發薪

admin 選擇：

- 資金帳戶
- 發薪日期
- idempotency key

系統為每位實發薪資大於 0 的員工建立一筆：

```text
Money Entry
方向：expense
分類：薪資 / 佣金
vehicle_id：null
金額：該員工實發薪資
對象：員工姓名
來源：salary_settlement
approval_status：approved
```

因為只有 admin 可執行發薪，所以建立後直接 approved，並立即反映資金帳戶餘額與 Dashboard 支出。

每位員工每個月份只能對應一筆正式薪資支出。重試相同 payload 必須 replay；同一 key 不同 payload 必須拒絕；並發重複發薪不得產生兩筆 Money Entry。

已發薪後不得直接修改或刪除原始結算與薪資支出。若需補發或扣回，於後續月份建立有原因的調整項目，或由 admin 依既有一般收支規則另建修正紀錄。

## 9. 權限與隱私

v1.3 初版採最小權限：

| 功能 | admin | manager | sales |
|---|---:|---:|---:|
| 查看全公司薪資結算 | 是 | 否 | 否 |
| 設定薪資資料 | 是 | 否 | 否 |
| 管理獎金方案 | 是 | 否 | 否 |
| 建立／重算月份草稿 | 是 | 否 | 否 |
| 新增手動加扣項 | 是 | 否 | 否 |
| 確認結算 | 是 | 否 | 否 |
| 執行發薪 | 是 | 否 | 否 |
| 查看自己的薪資單 | 本版不做 | 本版不做 | 本版不做 |

manager／sales 不得透過：

- API
- 前端路由
- Sidebar
- Dashboard
- ID 枚舉

取得任何他人或全公司的底薪、勞健保扣款、獎金或實發薪資。

未來若需要員工自助查詢自己的薪資單，另立小版本並建立 owner-only Resource，不在 v1.3 首版範圍。

- `id`
- `commission_plan_id`
- `min_sales_count`

- `sort_order`
- timestamps

初始資料：

```text
min_sales_count=1  → 2000 bps

```

### 10.4 `vehicles` 新增欄位

- `purchase_agent_id`

### 10.6 `salary_settlements`

- `gross_pay`
- `deduction_total`

- timestamps

### 10.7 `salary_settlement_items`

- `id`
- `salary_settlement_id`
- `type`：底薪／固定津貼／收車獎金／賣車獎金／勞保／健保／其他加給／其他扣款

- `amount`
- `description`
- `calculation_snapshot`，JSON nullable
- `created_by`
- timestamps

車輛獎金項目的 snapshot 至少需保存：

- 單車 approved 收入
- 單車 approved 支出
- 單車毛利
- 公司保留比例與金額
- 分配池
- 收車比例與金額
- 賣車比例與金額
- 公司剩餘分配額
- 收車人／賣車人
- 所屬月份

5. 待補收車人／賣車人與資料異常清單

薪資月份詳情需顯示：

- 全公司預估實發合計
- 每位員工底薪、獎金、加扣項與實發
- 每位員工成交台數、目前級距與下一級距
- 每台車獎金明細
- 公司營運保留與公司剩餘分配額摘要
- 阻擋確認的異常
- 確認結算按鈕
- 發薪按鈕

## 12. 稽核與資料安全

v1.3 需寫入 Audit Log：

- 薪資設定新增／修改
- 獎金方案建立
- 薪資月份建立／重算
- 手動加扣項新增／刪除
- 薪資確認
- 發薪

Audit Log 不應直接存完整薪資內容或敏感明細快照，避免稽核紀錄成為另一份薪資資料外洩面；只記錄必要的模型、月份、操作者與動作結果。

API Resource 必須採白名單，不得直接序列化完整 Model。

## 14. 成功標準

v1.3 完成後必須達成：

```text
老闆選擇月份
→ 系統自動找出該月符合資格的成交車
→ 依每位賣車人的整月成交台數套用跨級比例
→ 正確計算每台車收車獎金、賣車獎金與公司保留
→ 自動彙總底薪、津貼、勞健保與手動加扣項
→ 顯示每位員工實發薪資與全公司應準備金額
→ 老闆確認後鎖定
→ 發薪時每位員工只建立一筆正式薪資支出
→ 重試或並發不會重複發薪
```

同時必須保證：

- pending／rejected 收支不影響毛利與獎金
- 歷史規則與薪資快照不可被新設定回改
- 缺少收車人／賣車人不可靜默略過
- manager／sales 不能讀取薪資 API
- 已發薪不可直接修改或刪除
- 所有金額公式皆使用整數，總額不超配、不漏一元
