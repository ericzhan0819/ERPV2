# ERPV2 v1.1 企劃書：實務工作流補強

本文件是 `企劃書.md` 的 v1.1 延伸文件，需與原企劃書並讀，不取代原本 1.0 規格。

ERPV2 1.0 已完成中古車行內部營運系統的基礎工程 MVP：登入、Dashboard、車輛管理、車輛流程、收支、資金帳戶、使用者、列印與文件。v1.1 的目標不是重做系統，而是把 1.0 從「能運作的工程骨架」補強成更貼近中古車行日常操作的實務工作流。

---

## 1. v1.1 產品定位

v1.1 定位為：

```text
ERPV2 實務工作流補強版
```

核心目標：

```text
補強入庫建檔、購車付款、客戶管理、員工帳號、固定角色權限與一般收支審核，使系統更接近中古車行日常作業。
```

v1.1 仍維持 1.0 的產品邊界：

```text
不做正式會計
不做借方 / 貸方
不做會計科目
不做傳票
不做報稅
不做發票
不做 POS
不做完整 ERP 權限勾選矩陣
不做 HR / 薪資 / 打卡 / 排班
不做附件 / OCR / QR Code
不做記帳士交接包
```

## 3. 權限設計：從 is_admin 升級為固定角色

## 3.1 設計原則

1.0 使用 `is_admin` / `is_active` 作為簡化權限基礎。v1.1 起新增 `users.role`，並以 `role` 作為唯一正式權限來源。

`is_admin` 欄位在 v1.1 保留，但不再作為權限判斷依據。它只作為舊版相容與 migration 安全網。v1.2 之後再評估是否移除。

---

## 3.2 角色

v1.1 固定三種角色：

```text
admin    管理員 / 老闆 / 最高權限
manager  經理 / 主管 / 營運管理
sales    業務 / 銷售執行
```

不做完整角色權限勾選 UI。未來若需要新增 `accounting` 或 `staff`，可透過資料庫欄位與後端矩陣擴充，但 v1.1 不實作。

---

## 3.3 既有帳號回填規則

新增 `users.role` 後，既有帳號依下列規則回填：

```text
is_admin = true  → role = admin
is_admin = false → role = manager
```

理由：1.0 中 `is_admin=false` 的帳號，除了使用者管理與資金帳戶寫入外，其餘車輛、收支、收購價、毛利等資料多數可見。若直接回填為 `sales`，會造成既有帳號權限突然下降，屬於破壞性變更。回填為 `manager` 較能保留既有能見度，再由 admin 手動調整為 sales。

---

## 5. 建車同步購車付款

## 5.1 現況問題

1.0 流程為：

```text
建立車輛
→ 進車輛詳情
→ 另行記錄購車付款
```

這與實際入庫時常見的「建檔同時付款」不一致。

---

## 5.2 設計原則

收購價不等於實際已付款金額。

例如：

```text
收購價 500,000
今日先付 100,000
剩餘尾款之後支付
```

因此 v1.1 不可因填寫 `purchase_price` 就自動扣款。必須由使用者在建車表單中明確選擇是否同步建立購車付款。

---

## 5.3 表單欄位

新增車輛表單的「購車付款」區塊包含：

```text
是否建立時同步記錄購車付款
付款金額
付款帳戶
付款日期
付款備註
```

若未勾選同步付款，系統只建立車輛，不建立 MoneyEntry。

若勾選同步付款，後端在同一個 transaction 中建立：

```text
Vehicle
MoneyEntry：
  direction = expense
  category = 購車付款
  vehicle_id = 新車輛 ID
  source_type = vehicle_workflow 或明確定義的新 workflow source
```

---

## 5.4 權限

只有以下角色可在建車時同步購車付款：

```text
admin
manager
```

`sales` 不可新增車輛，也不可觸發建車同步購車付款。

---

## 5.5 冪等設計

新增車輛 API 需支援 `idempotency_key`。

建議新增：

```text
vehicles.idempotency_key nullable unique
```

歷史資料為 `null`；MySQL unique index 允許多筆 `null`。

建車同步付款需沿用既有 VehicleService / MoneyEntryService 的冪等模式：

```text
unique idempotency_key
payload 比對
相同 payload replay
不同 payload reject
QueryException race 後在新交易中 lockForUpdate 重讀
```

initial_purchase_payment 的 MoneyEntry idempotency key 可由 vehicle key 衍生，例如：

```text
{vehicle_idempotency_key}:initial-payment
```

前端不需要額外生成第二把 key。

---

## 5.6 Transaction 規則

建車同步購車付款必須符合：

```text
建立車輛與建立付款在同一個 DB transaction
付款帳戶停用時，整個交易回滾
付款金額不合法時，整個交易回滾
付款建立失敗時，車輛不得殘留
```

必須測試「付款失敗時車輛不存在」。

## 8. 一般收支審核流程

## 8.1 目標

v1.1 允許 manager / sales 送出一般收支申請，例如公務支出、出差費、臨時代墊。但這些收支必須經 admin 核准後，才正式計入資金餘額。

這不是正式會計簽核，也不是傳票流程。

---

## 8.2 money_entries 新增欄位

money_entries 新增：

```text
approval_status
approved_by nullable
approved_at nullable
```

approval_status：

```text
approved
pending
rejected
```

既有資料一律回填為：

```text
approved
```

避免 v1.1 上線後追溯影響既有餘額。

---

## 8.3 建立規則

```text
admin 建立的一般收支 → approval_status = approved
manager 建立的一般收支 → approval_status = pending
sales 建立的一般收支 → approval_status = pending
```

前端不得傳入或偽造 `approval_status`。狀態必須由後端依目前登入使用者 role 決定。

---

## 8.4 核准 / 駁回規則

新增 admin-only 審核操作：

```text
核准 pending 收支
駁回 pending 收支
```

建議 API：

```text
PATCH /api/money-entries/{id}/approve
PATCH /api/money-entries/{id}/reject
```

規則：

```text
只有 admin 可核准 / 駁回
manager 不可核准
sales 不可核准
approved/rejected 後不可再改回 pending
rejected 保留紀錄但不計入餘額
若要修正 rejected 或 approved 紀錄，需建立新 entry，不做狀態回復
```

---

## 8.5 餘額計算規則

所有正式餘額與金額彙總，只能計入：

```text
approval_status = approved
```

需影響：

```text
Dashboard 資金與本月收支
CashAccount balances
Vehicle 單車收支摘要
MoneyEntryService::balanceForAccount()
MoneyEntryService::balanceForType()
```

pending / rejected 不可影響正式餘額。

## 12. v1.1 Smoke 驗收流程

v1.1 完成後需人工 smoke：

```text
1. admin 登入
2. 建立完整入庫車輛
3. 建車時同步建立購車付款
4. 車輛詳情看到購車付款
5. 單車支出 / 毛利摘要正確
6. 新增整備支出
7. 整備完成上架
8. 建立客戶
9. 保留車輛時選買方客戶
10. 收訂金
11. 收尾款
12. 成交結案
13. 列印成交結案收支明細
14. sales 帳號登入，確認看不到收購價 / 毛利 / 資金
15. sales 可操作保留 / 收尾款 / 成交結案
16. sales 不可新增車輛 / 上架 / 設定定價
17. sales 可送出一般收支申請，但不影響餘額
18. manager 可操作車輛建檔 / 上架 / 客戶管理
19. manager 送出一般收支仍需 admin 核准
20. admin 可核准 / 駁回一般收支
21. approved 後才影響 Dashboard / CashAccount / Vehicle 摘要
22. rejected 不影響餘額且保留紀錄
```

## 14. 實作順序建議

v1.1 建議順序：

```text
1. users role / 員工帳號欄位 schema
2. role-based Policy / Middleware / Resource 遮蔽
3. 一般收支審核流程
4. Customer Module
5. 車輛入庫建檔欄位與表單
6. 建車同步購車付款
7. UI/UX 收斂與完整 smoke
```

理由：

```text
角色與欄位基礎先定，後續 Customer、Vehicle、MoneyEntry 才能依角色矩陣設計。
一般收支審核會動到餘額核心計算，需早點隔離測試。
建車同步購車付款同時牽涉建車、付款、transaction、idempotency 與角色，應排在後段，等前置基礎穩定後再做。
```

## 16. Codex review 風險重點

v1.1 需特別注意下列可能被 review 挑戰的點：

```text
role 回填是否造成破壞性權限變更
sales 是否能透過 API 取得被前端隱藏的敏感欄位
pending money entries 是否被餘額計算誤納入
approved/rejected 狀態是否可被非法反轉
建車同步付款的 transaction 是否可能殘留半成品
建車 idempotency replay 是否會錯誤重建 MoneyEntry
同 idempotency key 不同 payload 是否確實 422
QueryException race 是否在新交易中重讀
Customer 快照欄位是否避免歷史資料被客戶異動污染
manager 權限是否越權管理使用者或現金帳戶
```

## 17. 文件關係
